> 후니프린팅 shopby Enterprise 기반 주문/결제/파일관리 시스템 (15개 기능, 6개 모듈)

## 1. 개요

### 1.1 목적

후니프린팅 shopby Enterprise 마이그레이션에서 주문관리(A6B8-ORDER) 도메인의 전체 기능을 정의한다. 인쇄 파일 업로드/검수, 옵션보관함, 결제, 주문관리, 파일확인, 주문상태 트래킹 기능을 포괄하며, shopby NATIVE/SKIN과 CUSTOM 개발을 혼합하는 Hybrid 전략을 따른다.

### 1.2 범위

- **포함**: 파일/편집 정보입력, 보관함/장바구니, 배송정보입력, 배송지목록, 결제(이니시스 연동), 주문완료, 주문관리(인쇄/제본/굿즈), 파일확인처리, 재업로드요청, 주문서출력, 주문상태변경, 주문상태변경(일괄), 후불결제, 증빙서류발급, 고객 SMS 발송
- **제외**: 상품옵션/가격 엔진(SPEC-PRODUCT), 원장관리(SPEC-LEDGER), 체험단(SPEC-PROMOTION), 팀별 통계(SPEC-STATS)

### 1.3 SPEC-PLAN-001과의 관계

본 SPEC은 SPEC-PLAN-001 v1.1.0 마스터 기획서의 A6B8-ORDER 도메인 정책을 구현 수준으로 구체화한 문서이다. 정책 체크리스트의 A-6-1 ~ A-6-6, B-8-1 ~ B-8-10 항목을 포괄한다.

### 1.4 도메인 특성

- **Hybrid 접근**: Tier 1 (NATIVE/SKIN) 10개 + Tier 2 (CUSTOM) 5개 기능 혼합
- **CUSTOM 개발 핵심**: 인쇄 파일 업로드/검수(XL), 파일확인처리(XL), 후불결제(L), 재업로드요청(M), 고객 SMS(M)
- **외부 의존성**: KG이니시스 PG, 네이버페이, 카카오 알림톡, PitStop Server, AWS S3
- **인쇄 공정 연동**: 8단계 인쇄 워크플로우와 주문 상태 동기화

## 4. EARS 요구사항

### 4.1 모듈 1: 파일 업로드/정보입력 (File Upload) - A-6-1

#### REQ-ORDER-001 [Ubiquitous] 인쇄 파일 업로드 필수

시스템은 항상 인쇄 상품 주문 시 인쇄 파일 업로드를 필수로 요구해야 한다. (굿즈 상품 제외)

#### REQ-ORDER-002 [Ubiquitous] 파일 포맷 제한

시스템은 항상 PDF, AI, PSD, JPG, PNG 포맷만 업로드를 허용해야 한다.

#### REQ-ORDER-003 [Ubiquitous] 파일 크기 제한

시스템은 항상 단일 파일 최대 300MB, 주문당 총 1GB 이하로 파일 크기를 제한해야 한다.

#### REQ-ORDER-004 [Event-Driven] 드래그앤드롭 업로드

WHEN 사용자가 파일을 업로드 영역에 드래그앤드롭하면 THEN 시스템은 파일을 S3에 업로드하고 진행률을 실시간으로 표시해야 한다.

#### REQ-ORDER-005 [Event-Driven] 클라이언트 기본 검증

WHEN 파일이 업로드되면 THEN 시스템은 클라이언트 사이드에서 포맷, 파일 크기, 파일명 유효성을 즉시 검증해야 한다.

#### REQ-ORDER-006 [Event-Driven] 서버 상세 검증 (PitStop)

WHEN 파일이 S3에 업로드 완료되면 THEN 시스템은 PitStop Server에 프리플라이트 검증을 요청하고 결과를 3단계(통과/경고/오류)로 표시해야 한다.

#### REQ-ORDER-007 [Event-Driven] 검증 결과 표시

WHEN PitStop 검증이 완료되면 THEN 시스템은 해상도(300DPI), 컬러모드(CMYK), 도련(3mm), 폰트 임베딩 결과를 항목별로 표시해야 한다.

#### REQ-ORDER-008 [State-Driven] 경고 상태 진행 허용

IF 파일 검증 결과가 '경고' 상태이면 THEN 시스템은 사용자에게 경고 내용을 고지한 후 동의 시 주문 진행을 허용해야 한다.

#### REQ-ORDER-009 [State-Driven] 오류 상태 진행 차단

IF 파일 검증 결과가 '오류' 상태이면 THEN 시스템은 주문 진행을 차단하고 재업로드를 안내해야 한다.

#### REQ-ORDER-010 [Event-Driven] PDF 템플릿 다운로드

WHEN 상품 상세 페이지에서 "인쇄 가이드라인 다운로드" 버튼을 클릭하면 THEN 시스템은 해당 상품별 PDF 템플릿 파일을 다운로드해야 한다.

#### REQ-ORDER-011 [Event-Driven] 편집 정보 입력

WHEN 파일 업로드 후 인쇄 옵션을 선택하면 THEN 시스템은 선택된 옵션 조합(용지, 수량, 후가공 등)을 파일과 함께 저장해야 한다.

#### REQ-ORDER-012 [Unwanted] 미검증 파일 주문 금지

시스템은 PitStop 검증을 거치지 않은 파일로 인쇄 상품 주문을 생성하지 않아야 한다.

#### REQ-ORDER-013 [Unwanted] 업로드 중 페이지 이탈 방지

시스템은 파일 업로드 진행 중 페이지 이탈 시 확인 다이얼로그를 표시하지 않고 이탈을 허용하지 않아야 한다.

### 4.2 모듈 2: 보관함/장바구니 (Option Storage) - A-6-2

#### REQ-ORDER-014 [Event-Driven] 옵션보관함 저장

WHEN 사용자가 인쇄 옵션 조합과 파일을 완성하고 "보관함에 저장" 버튼을 클릭하면 THEN 시스템은 해당 조합을 옵션보관함에 저장해야 한다.

#### REQ-ORDER-015 [Ubiquitous] 보관기간 제한

시스템은 항상 옵션보관함 항목을 저장일로부터 1개월(30일)간 보관해야 한다.

#### REQ-ORDER-016 [Event-Driven] 보관함 재주문

WHEN 옵션보관함에서 "재주문" 버튼을 클릭하면 THEN 시스템은 저장된 옵션 조합과 파일을 불러와 주문서를 자동 구성해야 한다.

#### REQ-ORDER-017 [State-Driven] 만료 항목 자동 삭제

IF 보관함 항목이 30일 경과하면 THEN 시스템은 해당 항목과 연결된 파일을 자동 삭제해야 한다.

#### REQ-ORDER-018 [Event-Driven] 장바구니 연동

WHEN 인쇄 옵션 조합을 장바구니에 추가하면 THEN 시스템은 shopby 장바구니 API와 커스텀 옵션 스토리지에 동시에 데이터를 저장해야 한다.

#### REQ-ORDER-019 [Unwanted] 파일 없는 장바구니 금지

시스템은 인쇄 상품을 파일 업로드 없이 장바구니에 추가하지 않아야 한다. (굿즈 제외)

### 4.3 모듈 3: 배송/결제 (Shipping & Payment) - A-6-3 ~ A-6-6

#### REQ-ORDER-020 [Ubiquitous] 무료배송 기준

시스템은 항상 주문 금액 50,000원 이상 시 무료배송을 적용해야 한다.

#### REQ-ORDER-021 [Ubiquitous] 기본 배송비

시스템은 항상 주문 금액 50,000원 미만 시 3,000원의 기본 배송비를 부과해야 한다.

#### REQ-ORDER-022 [Ubiquitous] 제주 추가 배송비

시스템은 항상 제주도 배송 시 3,000원의 추가 배송비를 부과해야 한다.

#### REQ-ORDER-023 [Ubiquitous] 도서산간 추가 배송비

시스템은 항상 도서산간 지역 배송 시 5,000원의 추가 배송비를 부과해야 한다.

#### REQ-ORDER-024 [State-Driven] 배너/현수막 배송비 별도

IF 주문에 배너/현수막 상품이 포함되면 THEN 시스템은 해당 상품에 별도 배송비 템플릿을 적용해야 한다. (무료배송 기준 제외)

#### REQ-ORDER-025 [State-Driven] 혼합주문 배송비

IF 서로 다른 배송비 그룹의 상품이 혼합 주문되면 THEN 시스템은 최대 배송비를 1건만 부과해야 한다.

#### REQ-ORDER-026 [Event-Driven] 배송지 목록 조회

WHEN 주문서 작성 화면에서 배송지를 선택하면 THEN 시스템은 회원의 저장된 배송지 목록을 표시하고 기본 배송지를 자동 선택해야 한다.

#### REQ-ORDER-027 [Event-Driven] 배송지 신규 입력

WHEN "새 배송지 입력" 버튼을 클릭하면 THEN 시스템은 다음 우편번호 검색 API를 통해 주소를 입력받아야 한다.

#### REQ-ORDER-028 [Event-Driven] KG이니시스 결제 실행

WHEN 사용자가 결제 수단을 선택하고 "결제하기" 버튼을 클릭하면 THEN 시스템은 KG이니시스 PG 결제 모듈을 호출해야 한다.

#### REQ-ORDER-029 [Event-Driven] 네이버페이 결제

WHEN 네이버페이 결제를 선택하면 THEN 시스템은 네이버페이 결제 모듈을 호출하고 결제 완료 콜백을 처리해야 한다.

#### REQ-ORDER-030 [Event-Driven] 결제 실패 팝업 안내

WHEN 결제가 실패하면 THEN 시스템은 즉시 팝업으로 실패 사유를 안내하고 재시도 버튼을 제공해야 한다.

#### REQ-ORDER-031 [Event-Driven] 결제 실패 알림톡

WHEN 결제가 실패하면 THEN 시스템은 카카오 알림톡으로 결제 실패 안내와 재결제 링크를 발송해야 한다.

#### REQ-ORDER-032 [Event-Driven] 주문 완료 확인

WHEN 결제가 성공하면 THEN 시스템은 주문 완료 페이지를 표시하고 주문번호, 결제 금액, 예상 제작일을 안내해야 한다.

#### REQ-ORDER-033 [Event-Driven] 주문 완료 메일 발송

WHEN 주문이 완료되면 THEN 시스템은 주문 확인 이메일을 자동 발송해야 한다.

#### REQ-ORDER-034 [Event-Driven] 주문 완료 알림톡

WHEN 주문이 완료되면 THEN 시스템은 카카오 알림톡으로 주문번호와 예상 제작일을 발송해야 한다.

#### REQ-ORDER-035 [Ubiquitous] 무이자할부 기본 제공

시스템은 항상 2~3개월 무이자할부 옵션을 기본 제공해야 한다. (카드사 이벤트에 따라 월별 변경)

### 4.4 모듈 4: 주문관리 (Order Management) - B-8-1, B-8-6, B-8-7, B-8-7b

#### REQ-ORDER-036 [Ubiquitous] 인쇄 공정 8단계 상태 코드

시스템은 항상 주문 상태를 8단계로 관리해야 한다: ORDER_RECEIVED(접수) -> FILE_CHECKING(파일확인) -> PRINT_READY(인쇄준비) -> PRINTING(인쇄중) -> POST_PROCESSING(후가공) -> PACKAGING(포장) -> SHIPPED(출고) -> DELIVERING(배송중).

#### REQ-ORDER-037 [Event-Driven] 주문목록 조회 (관리자)

WHEN 관리자가 주문관리 페이지에 접근하면 THEN 시스템은 주문 목록을 상태별/기간별/상품유형별(인쇄/제본/굿즈)로 필터링 가능하게 표시해야 한다.

#### REQ-ORDER-038 [Event-Driven] 주문 상세 조회 (관리자)

WHEN 관리자가 주문 상세를 조회하면 THEN 시스템은 주문 정보, 인쇄 옵션, 업로드 파일, 현재 공정 상태, 결제 정보를 한 화면에 표시해야 한다.

#### REQ-ORDER-039 [Event-Driven] 주문상태 변경

WHEN 관리자가 주문 상태를 변경하면 THEN 시스템은 shopby 주문 상태를 업데이트하고 커스텀 인쇄 공정 상태를 동기화해야 한다.

#### REQ-ORDER-040 [Event-Driven] 상태 변경 알림

WHEN 주문 상태가 변경되면 THEN 시스템은 고객에게 카카오 알림톡으로 상태 변경 안내를 발송해야 한다.

#### REQ-ORDER-041 [Event-Driven] 주문상태 일괄 변경

WHEN 관리자가 다수 주문을 선택하고 "일괄 상태변경" 버튼을 클릭하면 THEN 시스템은 선택된 모든 주문의 상태를 동시에 변경해야 한다.

#### REQ-ORDER-042 [Event-Driven] 주문서 출력

WHEN 관리자가 "주문서 출력" 버튼을 클릭하면 THEN 시스템은 인쇄용 주문서(주문 정보 + 인쇄 사양 + 파일명)를 PDF로 생성해야 한다.

#### REQ-ORDER-043 [Unwanted] 역방향 상태 변경 금지

시스템은 인쇄 공정 상태를 역방향으로 변경하지 않아야 한다. (PRINTING -> FILE_CHECKING 등 불가, 예외: 관리자 강제 롤백)

### 4.5 모듈 5: 파일확인/재업로드 (File Review) - B-8-4, B-8-5

#### REQ-ORDER-044 [Event-Driven] 파일 검수 대기 목록

WHEN 관리자가 파일확인 페이지에 접근하면 THEN 시스템은 FILE_CHECKING 상태의 주문 목록을 접수 시간순으로 표시해야 한다.

#### REQ-ORDER-045 [Event-Driven] 파일 프리뷰

WHEN 관리자가 업로드 파일을 선택하면 THEN 시스템은 파일 미리보기(썸네일)와 PitStop 검증 결과를 함께 표시해야 한다.

#### REQ-ORDER-046 [Event-Driven] 파일 검수 승인

WHEN 관리자가 파일 검수를 "승인"하면 THEN 시스템은 주문 상태를 PRINT_READY로 변경하고 고객에게 "제작 시작" 알림톡을 발송해야 한다.

#### REQ-ORDER-047 [Event-Driven] 재업로드 요청

WHEN 관리자가 파일 검수를 "재업로드 요청"하면 THEN 시스템은 문제 내용(사유 선택/직접 입력)과 재업로드 링크를 포함한 알림톡+SMS를 고객에게 발송해야 한다.

#### REQ-ORDER-048 [Event-Driven] 고객 재업로드 처리

WHEN 고객이 재업로드 링크를 통해 새 파일을 업로드하면 THEN 시스템은 PitStop 재검증을 수행하고 관리자에게 "재업로드 접수" 알림을 발송해야 한다.

#### REQ-ORDER-049 [State-Driven] 재업로드 대기 타임아웃

IF 재업로드 요청 후 48시간 동안 재업로드가 없으면 THEN 시스템은 관리자에게 리마인더 알림을 표시하고 고객에게 리마인드 SMS를 재발송해야 한다.

#### REQ-ORDER-050 [Unwanted] 검수 건너뛰기 금지

시스템은 인쇄 상품의 파일 검수 단계를 건너뛰고 인쇄 준비 상태로 진행하지 않아야 한다.

### 4.6 모듈 6: 후불결제/증빙/SMS (B2B & Communications) - B-8-8 ~ B-8-10

#### REQ-ORDER-051 [State-Driven] 후불결제 주문 생성

IF 주문자가 B2B 거래처(후불 승인 등급)이면 THEN 시스템은 결제 없이 주문을 생성하고 '미결제' 상태로 기록해야 한다.

#### REQ-ORDER-052 [Event-Driven] 후불결제 정산 처리

WHEN 관리자가 후불결제 주문을 "결제 완료" 처리하면 THEN 시스템은 입금 확인일과 결제 수단을 기록하고 상태를 '결제완료'로 변경해야 한다.

#### REQ-ORDER-053 [Event-Driven] 미결제 목록 조회

WHEN 관리자가 후불결제 관리 페이지에 접근하면 THEN 시스템은 미결제/결제완료 상태별로 거래처 주문 목록을 표시해야 한다.

#### REQ-ORDER-054 [Unwanted] 미승인 후불결제 차단

시스템은 후불결제 승인 등급이 없는 고객의 후불 주문을 생성하지 않아야 한다.

#### REQ-ORDER-055 [Event-Driven] 증빙서류 발급 (세금계산서)

WHEN 고객 또는 관리자가 세금계산서 발급을 요청하면 THEN 시스템은 팝빌 API를 통해 전자세금계산서를 발급하고 이메일로 전송해야 한다.

#### REQ-ORDER-056 [Event-Driven] 증빙서류 발급 (현금영수증)

WHEN 현금결제 시 현금영수증 발급을 요청하면 THEN 시스템은 팝빌 API를 통해 현금영수증을 발급해야 한다.

#### REQ-ORDER-057 [Event-Driven] 고객 SMS 개별 발송

WHEN 관리자가 주문 상세에서 "SMS 발송" 버튼을 클릭하면 THEN 시스템은 템플릿 선택/직접 입력 방식으로 고객에게 SMS를 발송해야 한다.

#### REQ-ORDER-058 [Event-Driven] 고객 SMS 일괄 발송

WHEN 관리자가 다수 주문을 선택하고 "일괄 SMS" 버튼을 클릭하면 THEN 시스템은 선택된 주문의 고객들에게 동일한 메시지를 일괄 발송해야 한다.

#### REQ-ORDER-059 [Ubiquitous] 알림톡 우선 원칙

시스템은 항상 고객 알림 발송 시 카카오 알림톡을 우선 시도하고, 알림톡 수신 불가 시 SMS를 대체 발송해야 한다.

## 5. 비기능 요구사항

### 5.1 성능

#### REQ-ORDER-NF-001 [Ubiquitous] 파일 업로드 속도

시스템은 항상 100MB 파일 업로드를 30초 이내에 완료해야 한다. (멀티파트 업로드, 프로그레스바 표시)

#### REQ-ORDER-NF-002 [Ubiquitous] PitStop 검증 응답 시간

시스템은 항상 PDF 프리플라이트 검증 결과를 파일 크기 대비 30초 이내에 반환해야 한다.

#### REQ-ORDER-NF-003 [Ubiquitous] 결제 응답 시간

시스템은 항상 PG 결제 요청에서 결과 수신까지 5초 이내에 처리해야 한다.

### 5.2 보안

#### REQ-ORDER-NF-004 [Ubiquitous] 파일 접근 제어

시스템은 항상 업로드된 인쇄 파일을 해당 주문의 고객과 관리자만 접근할 수 있도록 제한해야 한다. (S3 Presigned URL)

#### REQ-ORDER-NF-005 [Ubiquitous] 결제 정보 비저장

시스템은 카드번호, CVC 등 결제 수단 정보를 서버에 저장하지 않아야 한다. (PG 위임)

#### REQ-ORDER-NF-006 [Ubiquitous] 주문 데이터 암호화

시스템은 항상 주문자 개인정보(이름, 연락처, 주소)를 전송 시 HTTPS, 저장 시 AES-256으로 암호화해야 한다.

### 5.3 가용성

#### REQ-ORDER-NF-007 [Ubiquitous] 결제 재시도

시스템은 항상 PG 결제 실패 시 최대 3회까지 자동 재시도를 지원해야 한다.

#### REQ-ORDER-NF-008 [Ubiquitous] 파일 업로드 이어받기

시스템은 항상 대용량 파일 업로드 중 네트워크 오류 발생 시 이어받기(Resume Upload)를 지원해야 한다.

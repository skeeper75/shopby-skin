## 1. 코드베이스 현황 분석

### 1.1 현재 페이지 구현 현황

| 영역 | 현황 | 비고 |
|------|------|------|
| 고객 페이지 | 40개 구현 완료 | 반응형 SPEC-LAYOUT-001/002 적용됨 |
| 관리자 페이지 | 17섹션 구조 존재 | Mock API 상태 |
| SPEC-SKIN-005 (관리자 기반/주문관리) | 80-85% UI 마이그레이션 | 백엔드 API 0% |

### 1.2 레이아웃 컴포넌트 (완성도 높음)

| 토큰 | 값 |
|------|-----|

### 1.4 파일 처리 워크플로우

**5단계 파이프라인:**

```
1. 파일 업로드 --> S3 --> SQS --> Job 생성 (STAT_010)
2. 문서 검증 --> PitStop 01
3. 핵심 처리 --> PitStop 02 (폰트 임베드, RGB->CMYK)
4. 평탄화 --> PitStop 03 (조건부)
5. 최종 처리 --> PDF 리사이즈, S3 업로드
```

**Job 상태 머신:**

| 코드 | 상태 | 설명 |
|------|------|------|
| STAT_010 | 접수 | 주문 접수 |
| STAT_020 | 업로드완료 | 파일 업로드 완료 |
| STAT_030 | 처리중 | PitStop 처리 진행 |
| STAT_040 | 이동중 | 파일 이동 |
| STAT_050 | 완료 | 처리 완료 |
| STAT_900 | 오류 | 처리 실패 |

**기술 스택:**
- Frontend: Next.js 14+
- Backend: NestJS, Bull Queue
- Database: PostgreSQL (Neon)
- Storage: AWS S3 (3개 버킷)
- 인쇄 처리: Enfocus PitStop Server

## 2. shopby API 역량 분석

### 2.1 API 지원 현황 매트릭스

| 요구사항 | 지원 수준 | 구현 방식 | 비고 |
|---------|---------|---------|------|
| 다중 의존 옵션 | 부분 지원 | `multiLevelOptions[]` + 프론트엔드 필터링 | 4단계 이상 의존성은 커스텀 |
| 파일 업로드/검증 | 미지원 | 완전 커스텀 (AWS S3 + PitStop) | shopby 자체 파일 저장소 없음 |
| 동적 가격 계산 | 지원 | `additional-discounts` API + 프론트 로직 | 복잡한 공식은 프론트에서 |
| 주문 상태 추적 | 지원 | 기본 상태 + 커스텀 필드 확장 | 인쇄 단계 매핑 필요 |
| B2B 외상 관리 | 부분 지원 | Server API 확장 또는 마이크로서비스 | 별도 DB 필요 |
| 선금 충전 | 가능 | MyPay + 잔액 관리 커스텀 | PG 연동 |
| 반품/교환 | 지원 | Claim API | 자동 처리 가능 |
| 회원등급 & B2B | 지원 | `member-grades` API + 사업자 검증 | - |
| 쿠폰/프로모션 | 지원 | Promotion API | 자동 할인 |

### 2.2 핵심 API 엔드포인트 (Shop API)

| 주문 | `GET /cart` | 장바구니 조회 |
| 주문 | `POST /cart` | 장바구니 추가 |

| 주문 | `POST /payments/reserve` | 결제 예약 |
| 카테고리 | `GET /categories` | 카테고리 목록 (최대 5 depth) |
| 회원 | `GET /config/member-extra-info` | 회원 추가정보 설정 |

## 3. 해외 인쇄/이커머스 벤치마크

### 3.1 Product Configurator 패턴

| 패턴 | 대표 서비스 | 특징 |
|------|-----------|------|
| 단계형 (Wizard) | Vistaprint, MOO | 옵션 순서대로 선택, 진행률 표시, 인지 부하 감소 |
| 단일 페이지 | Printful, Shopify | 전체 옵션 즉시 노출 |
| 하이브리드 | Canva Print | 디자인 에디터 통합 후 스펙 선택 |
| Progressive Disclosure | - | 기본 옵션 노출 --> 고급 옵션 "더 보기" 토글 |

**가격 표시 관련:**
- Baymard Institute 조사: 사용자 86%가 단가 미제공에 불만
- 실시간 단가 표시 = 차별화 포인트

### 3.2 파일 업로드/에디터 통합

- 드래그앤드롭 존 + 실시간 파일 검증 (해상도/블리드/CMYK)
- 하이브리드 진입점: "디자인 있음" --> 업로드 직행, "디자인 없음" --> 템플릿/의뢰
- MOO: 브랜드 에셋 중앙 저장, 버전 관리

### 3.3 장바구니 & B2B 패턴

**장바구니:**
- 인쇄 사양 전체 표시 (미리보기 + 크기 + 용지 + 마감 + 단가)
- MOO: 3시간 주문 수정 창, "같은 사양으로 재주문" 원클릭

**B2B:**
- PO 결제, Net Terms
- 예산 한도, 다단계 승인 워크플로우

### 3.4 주문 추적 & 대시보드

**생산 단계 타임라인:**
```
접수 --> 파일검토 --> 인쇄준비 --> 인쇄중 --> 후처리 --> 포장 --> 발송 --> 배송
```

**디자인 라이브러리:**
- 그리드/리스트 뷰, 태그 검색, 폴더 분류

**선불 크레딧:**
- 상단 우측 잔액 표시, 대량 충전 할인, 자동 충전 옵션

### 3.5 필수 UI 컴포넌트 패턴

## 4. 한국 온라인 인쇄 사이트 벤치마킹

> 조사 대상: 레드프린팅, 와우프레스, 비즈하우스, 프린트시티, 씨프린팅

### 4.1 상품 옵션 선택 화면 공통 구조

```
규격/사이즈 --> 인쇄도수 --> 용지 선택 --> 코팅/후가공 --> 수량 --> 추가옵션
```

**핵심 발견:**

| 사이트 | 특징 |
|--------|------|
| 와우프레스 | 가장 상세한 옵션 구조 (규격, 도수, 용지, 후가공, 당일출고) |
| 비즈하우스 | 미리캔버스 에디터 연동 (디자인 먼저 --> 옵션 선택) |
| 공통 | 가격 실시간 계산 표시 (모든 사이트) |

**옵션 의존성 처리:**
- 상위 옵션 변경 시 하위 옵션 초기화 + 안내 문구 표시
- 모든 사이트에서 동일 패턴 적용

### 4.2 주문 플로우 2가지 패턴

| 패턴 | 플로우 | 대표 사이트 |
|------|--------|-----------|
| 파일 업로드형 | 상품 --> 옵션 --> 파일업로드 --> 결제 | 레드프린팅, 와우프레스 |
| 디자인 연동형 | 에디터 --> 디자인 --> 옵션 --> 장바구니 --> 결제 | 비즈하우스 |

**파일 요구사항 (공통):**
- PDF 권장
- CMYK 색상 모드
- 300dpi 이상
- 재단 + 도련 3mm
- 서체 아웃라인 처리

### 4.3 인쇄 특화 주문 상태

```
주문완료 --> 파일접수 --> 파일검수중 --> 제작접수 --> 인쇄중 --> 후가공 --> 출고 --> 배송중
```

**"주문 보류" 상태 (핵심 발견):**
- 파일 오류 발견 시 자동으로 "주문 보류" 상태 전환
- 카카오 알림톡으로 고객에게 즉시 발송
- 고객이 파일 수정 후 "제작시작" 버튼을 눌러야 진행
- 후니프린팅 Job 상태 STAT_900(오류)과 매핑 가능

### 4.4 경쟁사 대비 차별화 방향

| 항목 | 경쟁사 현황 | 후니프린팅 차별화 방향 |
|------|-----------|---------------------|
| 옵션 선택 | 텍스트 중심 나열 | 시각적 미리보기 강화 (용지/코팅 샘플 이미지) |
| 파일 검수 | 수동/지연 (수 시간 소요) | PitStop 자동 검수 + 즉각 피드백 |
| 주문 상태 | 단순 배송 추적 | 인쇄 공정별 상세 표시 + WebSocket 실시간 |
| 재주문 | 단순 재주문 | 파일 + 옵션 세트 보관함 저장 |

### 4.5 출처

redprinting.co.kr, wowpress.co.kr, bizhows.com, miricanvas.com, swadpia.co.kr

## 6. 크로스 레퍼런스 및 설계 시사점

### 6.1 구현 유형별 화면 매핑

**NATIVE (shopby 기본) --> 스킨 커스터마이징으로 해결:**
- 로그인/회원가입, 카테고리, 장바구니, 결제, 클레임 등
- Aurora Skin Provider 활용 (ProductDetailProvider, CartProvider 등)

**SKIN (프론트엔드 커스터마이징) --> 레이아웃 컴포넌트 활용:**
- 기존 `PageShell`, `ResponsiveGrid`, `SplitLayout` 재사용
- 디자인 토큰 (breakpoints, padding, typography) 통일

**CUSTOM (별도 개발) --> 독립 모듈 설계 필요:**
- 파일 업로드/검수: S3 + PitStop 파이프라인 연동
- 종속옵션 + 가격: `multiLevelOptions` 확장 + 프론트 계산 엔진
- 프린팅머니: MyPay 커스텀 + PG 연동
- 주문상태 트래킹: Job 상태 머신 (STAT_010~STAT_900) --> UI 타임라인

### 6.2 벤치마크 --> 화면 설계 적용 포인트

| 벤치마크 인사이트 | 적용 화면 | 구현 방향 |

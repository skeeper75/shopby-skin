# Aurora Skin vs Headless shopby API: 기술 전략 의사결정 보고서

> 작성일: 2026-03-19
> SPEC: SPEC-SCREEN-001 (후니프린팅 리뉴얼 전체 화면 설계 마스터 플랜)
> 분석 대상: shopby Aurora React Skin vs Headless API 직접 소비

---

## 1. Executive Summary (결론)

**권장안: 현행 Hybrid 전략 유지 (Aurora Skin + CUSTOM 독립 모듈)**

현재 코드베이스 분석 결과, 전면 Headless 전환은 **비용 대비 효과가 낮다**. 핵심 근거는 다음과 같다:

1. **이미 구현된 자산이 크다**: 308개 파일, ~59,000줄의 JSX 코드가 Aurora Skin 기반으로 동작 중
2. **CSS 충돌이 예상보다 경미하다**: 전체 코드베이스에서 `!important` 사용이 7건(2개 파일)에 불과
3. **CUSTOM 기능은 이미 독립적이다**: 인쇄 특화 25개 기능은 shopby 컴포넌트와 무관하게 독립 모듈로 설계됨
4. **Headless 전환 시 60개 화면 재구축 필요**: NATIVE 35개 + SKIN 25개 화면을 처음부터 다시 만들어야 함

**결론: "고치지 않아도 되는 것을 고치지 마라"** - NATIVE/SKIN 화면은 Aurora Skin으로 빠르게 완성하고, 인쇄 특화 CUSTOM 모듈에 개발 역량을 집중하는 것이 최적 전략이다.

## 2. 정량 분석: 현재 코드베이스 현황

### 2.1 코드 규모

| 항목 | 수치 | 비고 |
|------|------|------|
| @shopby 임포트 파일 | 308개 | `@shopby/react-components` + `@shopby/shared` |
| 총 JSX/JS 코드 | ~59,000줄 | pages/ + components/ |
| `!important` 사용 | 7건 (2개 파일) | globals.css 6건, PrintSheet 1건 |
| 벤더 패키지 크기 | 1.2MB | .vendors/ 디렉토리 (tgz) |
| 구현 완료 고객 페이지 | 40개 | 반응형 SPEC-LAYOUT-001/002 적용 |
| Provider 사용 | 108건+ | 20개 파일에서 shopby Provider 활용 |

### 2.2 기능 분류별 현황 (95개 기능)

| 분류 | 개수 | 비율 | shopby 의존도 | Headless 전환 시 영향 |
|------|------|------|-------------|---------------------|
| NATIVE | 35개 | 40% | **높음** - Provider + API 래퍼 전면 사용 | 전면 재구축 필요 |
| SKIN | 25개 | 28% | **중간** - API 호출 + 부분 커스텀 UI | 대부분 재구축 필요 |
| CUSTOM | 25개 | 31% | **없음** - 완전 독립 모듈 | 영향 없음 |
| EXTERNAL | 2개 | 2% | **없음** - 외부 SDK | 영향 없음 |

### 2.3 CSS 오버라이드 실제 현황

`globals.css` 분석 결과, 오버라이드가 필요한 벤더 CSS 충돌은 **극히 제한적**이다:

```
globals.css:
  - .page-inner { max-width: 1280px !important; }
  - .page__content.site { width: 100% !important; max-width: 1280px !important; }
  - .page .bottom-nav { max-width: 1280px !important; }
  - .page .offcanvas__dim { max-width: 1280px !important; }
```

모두 **레이아웃 너비 확장** 관련이며, 기능적 충돌이 아닌 디자인 조정 수준이다. `@MX:WARN`과 `@MX:REASON` 태그로 문서화되어 있어 관리가 용이하다.

## 3. 상세 비교 분석

### 3.1 개발 속도 (Development Speed)

| 항목 | Aurora Skin (현행) | Headless (전환) | 판정 |
|------|-------------------|----------------|------|
| **초기 구축** | NATIVE 35개 기능 즉시 사용 | 60개 화면 처음부터 구축 | Skin 압도적 우위 |
| **CUSTOM 기능 개발** | 독립 모듈로 자유 구현 | 동일 (독립 모듈) | 동등 |
| **인쇄 Configurator** | shopby 옵션 API 부분 활용 가능 | 전체 자체 구현 | Skin 소폭 우위 |
| **총 프로젝트 일정** | Phase별 점진적 완성 가능 | 기본 기능 재구축으로 지연 | Skin 우위 |

**정량 추정:**

- Skin 유지: NATIVE 35개 화면은 스킨 래핑만으로 빠르게 완성 (화면당 평균 0.5~1일)
- Headless 전환: 로그인/회원가입/장바구니/결제 등 기본 플로우 재구축에 최소 4~6주 추가 소요
  - Auth 플로우 (OAuth2 + SNS): 1~2주
  - Cart + Order Sheet + Payment: 2~3주
  - 회원관리 (프로필, 주소, 쿠폰): 1~2주

### 3.2 유지보수성 (Maintainability)

| 항목 | Aurora Skin (현행) | Headless (전환) | 판정 |
|------|-------------------|----------------|------|
| **CSS 기술 부채** | 7건 !important, 추가 증가 가능하나 관리 가능 | 완전 자유 (기술 부채 없음) | Headless 우위 |
| **shopby 업데이트 호환성** | 벤더 패키지 업데이트 시 검증 필요 | API만 호환성 확인 | Headless 우위 |
| **디버깅 난이도** | Provider 체인 내부 동작 불투명 | 모든 코드 통제 가능 | Headless 우위 |
| **코드 가독성** | shopby 패턴 학습 필요 | 표준 React 패턴 | Headless 소폭 우위 |
| **팀 온보딩** | shopby 프레임워크 학습 곡선 있음 | 표준 Next.js + React | Headless 소폭 우위 |

**핵심 판단**: Headless가 유지보수에서 이론적으로 우위이나, 현재 CSS 충돌이 7건에 불과하고 모두 문서화되어 있어 **실질적 기술 부채가 낮다**.

### 3.3 기술적 자유도 (Technical Freedom)

| 항목 | Aurora Skin (현행) | Headless (전환) | 판정 |
|------|-------------------|----------------|------|
| **디자인 시스템** | PageShell/ResponsiveGrid로 독자 DS 운영 중 | 완전 자유 | 현행도 충분 |
| **반응형 최적화** | SPEC-LAYOUT-001/002로 달성 완료 | 동등 | 동등 |
| **인쇄 특화 기능** | CUSTOM 독립 모듈로 100% 자유 | 동일 | 동등 |
| **SSR/SSG/SEO** | Webpack SPA (SSR 불가) | Next.js App Router (SSR/SSG 가능) | **Headless 우위** |
| **번들 최적화** | 전체 @shopby 로드 (tree-shaking 제한) | 필요한 것만 로드 | **Headless 우위** |
| **React 버전** | React 18.2 (고정) | React 19+ (최신) | Headless 소폭 우위 |

**핵심 판단**: SSR/SEO가 비즈니스 크리티컬한지가 의사결정 분기점이다. 인쇄 업계 B2B 특성상 SEO 중요도는 상대적으로 낮다 (기존 고객 직접 접속 + 거래처 중심).

### 3.4 리스크 분석 (Risk Analysis)

#### Option A: Skin 유지 리스크

| 리스크 | 발생 확률 | 영향도 | 대응 방안 |

| shopby 메이저 업데이트 호환성 | 낮음 | 중간 | 벤더 패키지 버전 고정 (현재 tgz 관리) |
| Provider 내부 버그 | 낮음 | 높음 | Characterization 테스트로 감지, shopby 지원 요청 |
| 번들 사이즈 증가 | 중간 | 낮음 | 코드 스플리팅, 관리자 영역 별도 빌드 |

#### Option B: Headless 전환 리스크

| 리스크 | 발생 확률 | 영향도 | 대응 방안 |
|--------|---------|-------|----------|
| 일정 지연 (기본 기능 재구축) | **높음** | **높음** | 점진적 마이그레이션이나 여전히 상당한 지연 |
| 결제/인증 플로우 구현 오류 | 중간 | 높음 | PG 연동 테스트 강화, shopby API 문서 의존 |
| shopby API 미문서화 동작 발견 | 중간 | 중간 | API 탐색적 테스트 필수, 우회 방안 준비 |
| 두 시스템 병행 기간 복잡도 | 높음 | 중간 | 명확한 마이그레이션 단위 분리 |

#### Option C: Hybrid (현행 계획, 권장)

| 리스크 | 발생 확률 | 영향도 | 대응 방안 |
|--------|---------|-------|----------|
| CUSTOM/Skin 경계 모호 | 낮음 | 낮음 | SPEC 분류로 명확 구분 완료 |
| 두 가지 스타일 혼재 | 중간 | 낮음 | 관리자=Tailwind, 고객=Skin+커스텀 명확 분리 |

## 4. Trade-off Matrix (가중 평가)

| 기준 | 가중치 | Skin 유지 | Headless 전환 | Hybrid (권장) |
|------|--------|----------|-------------|--------------|
| 개발 속도 (초기) | 30% | 9 | 4 | 9 |
| 유지보수성 | 20% | 6 | 9 | 7 |
| 기술적 자유도 | 15% | 6 | 9 | 7 |
| 리스크 수준 (낮을수록 좋음) | 20% | 8 | 4 | 8 |
| 장기 확장성 | 15% | 5 | 9 | 6 |
| **가중 합계** | **100%** | **7.05** | **6.45** | **7.55** |

**Hybrid 전략이 최고 점수를 기록한다.** 개발 속도와 리스크 관리에서 압도적이며, 유지보수성과 자유도에서도 합리적 수준을 달성한다.

## 5. 권장 전략: Pragmatic Hybrid

### 5.1 전략 요약

```
쇼핑몰 NATIVE/SKIN (60개 화면)
  --> Aurora Skin 유지
  --> PageShell/ResponsiveGrid로 레이아웃 통일
  --> shopby Provider + API 래퍼 활용
  --> CSS 오버라이드는 @MX:WARN으로 관리

인쇄 특화 CUSTOM (25개 화면)
  --> 완전 독립 모듈
  --> Tailwind CSS + Radix UI (이미 도입됨)
  --> 커스텀 API (NestJS BFF + AWS S3 + PitStop)
  --> shopby 의존성 제로

관리자 백오피스 (35개 화면)
  --> Tailwind CSS + shadcn/ui 패턴
  --> AdminLayout 독자 체계 (SPEC-SKIN-005)
  --> shopby Server API 직접 호출
```

### 5.2 장기 전환 옵션 (Exit Strategy)

향후 shopby 의존도를 낮추고 싶다면 **점진적 Headless 전환** 경로가 열려 있다:

1. **Phase 1 (현재)**: Aurora Skin으로 빠르게 런칭
2. **Phase 2 (런칭 후 6개월)**: 트래픽 데이터 기반 SEO 필요성 재평가
3. **Phase 3 (필요시)**: 트래픽이 높은 페이지부터 Next.js SSR로 점진 전환
   - 상품 목록/상세 페이지 (SEO 영향 큰 페이지)
   - shopby API 직접 호출로 전환 (Provider 제거)
   - 나머지 페이지는 Skin 유지

이 접근법은 "지금 당장 전면 전환"의 리스크를 피하면서도 미래 유연성을 확보한다.

## 6. Risk Mitigation Plan

### 6.1 CSS 기술 부채 관리

| 조치 | 시점 | 효과 |
|------|------|------|
| `@MX:WARN` 태그 모든 !important에 적용 | 즉시 | 가시성 확보 |
| CSS `@layer` 도입 검토 | Phase 2 이후 | specificity 전쟁 예방 |
| 벤더 CSS 충돌 모니터링 스크립트 | Phase 1 완료 후 | 조기 감지 |
| !important 사용 10건 초과 시 경보 | 상시 | 기술 부채 한도 관리 |

### 6.2 shopby 벤더 리스크 관리

| 조치 | 시점 | 효과 |
|------|------|------|
| 벤더 패키지 tgz 버전 고정 | 현재 완료 | 예기치 않은 변경 방지 |
| Characterization 테스트 확대 | Phase 1 | Provider 동작 변경 감지 |
| shopby API 변경 모니터링 | 상시 | 호환성 이슈 조기 발견 |
| API 래퍼 레이어 유지 | 상시 | 향후 Headless 전환 시 API 호출부만 교체 |

### 6.3 번들 사이즈 관리

| 조치 | 시점 | 효과 |
|------|------|------|
| 코드 스플리팅 (React.lazy) | Phase 1 | 초기 로드 최적화 |
| 관리자/고객 빌드 분리 검토 | Phase 3 이후 | 고객 번들에서 관리자 코드 제거 |
| Bundle Analyzer 도입 | Phase 1 | 사이즈 모니터링 |

## 7. Cognitive Bias Check

분석의 객관성을 위해 인지 편향을 점검한다:

| 편향 | 점검 결과 |
|------|----------|
| **현상 유지 편향** (Status Quo Bias) | 인정: 현행 유지를 권장하므로 이 편향의 영향을 받을 수 있다. 그러나 정량 데이터(7건 !important, 60개 화면 재구축 비용)가 객관적으로 현행 유지를 지지한다. |
| **매몰 비용** (Sunk Cost) | 점검: 59,000줄의 기존 코드를 "아깝다"고 느끼는 것이 판단에 영향을 미치는가? -> 아니다. 기존 코드가 실제로 잘 동작하고 있고, 문제가 경미하기 때문이다. |
| **확증 편향** (Confirmation Bias) | 점검: Headless 전환의 장점(SSR, 번들 최적화, 유지보수)을 충분히 고려했는가? -> 그렇다. 3.3절과 3.2절에서 Headless의 실질적 우위를 인정했다. |
| **과신 편향** (Overconfidence) | 점검: CSS 충돌이 향후 증가하지 않을 것이라고 확신하는가? -> 아니다. 6.1절에서 모니터링과 경보 체계를 제안했다. |

**최종 판단**: 편향을 고려해도 권장안 변경 불필요. Headless 전환이 정당화되려면 **CSS 충돌이 50건 이상으로 증가하거나, SEO가 매출의 핵심 드라이버가 되거나, shopby가 서비스를 중단하는 시나리오**가 필요하다.

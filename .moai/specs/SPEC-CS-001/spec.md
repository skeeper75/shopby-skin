## 1. 개요

### 1.1 목적

후니프린팅 shopby Enterprise 마이그레이션에서 고객센터/관리자(A4B5-CS + B1-ADMIN) 도메인의 전체 기능을 정의한다. 공지사항, FAQ, 1:1문의, 비회원 주문조회, 이용후기관리, 체험단관리, 관리자 등록/관리 기능을 포괄하며, shopby NATIVE 게시판 기능을 최대한 활용하는 Tier 1 중심 구현 전략을 따른다.

### 1.2 범위

- **포함**: 공지사항(쇼핑몰+관리자), FAQ(쇼핑몰+관리자), 비회원 주문조회, 1:1문의(확인/답변), 체험단관리(CUSTOM), 이용후기관리, 관리자 등록/관리
- **제외**: 1:1문의 쇼핑몰 작성(SPEC-MYPAGE 또는 별도 SPEC), 이용후기 쇼핑몰 작성(SPEC-REVIEW 또는 상품상세 SPEC), 채팅 상담(P3 별도 SPEC)

### 1.3 SPEC-PLAN-001과의 관계

본 SPEC은 SPEC-PLAN-001 v1.1.0 마스터 기획서의 A4B5-CS 도메인(A-4-1, A-4-2, A-4-7, B-5-1, B-5-2, B-5-7, B-5-8, B-5-9) 및 B1-ADMIN 도메인(B-1-1) 항목을 구현 수준으로 구체화한 문서이다.

### 1.4 도메인 특성

| 구분 | 값 |
|------|---|
| 총 기능 수 | 9개 |
| NATIVE 비율 | 7/9 (78%) |

| 평균 복잡도 | 낮음 (S/M 위주) |

## 3. EARS 요구사항

### 3.1 모듈 1: 쇼핑몰 고객센터 (Customer Service Front) - 3개 기능

#### REQ-CS-001 [Ubiquitous] 공지사항 목록 기본 원칙

시스템은 항상 공지사항을 카테고리(공지/이벤트/상품)별로 분류하여 목록을 제공해야 한다.

#### REQ-CS-002 [Event-Driven] 공지사항 조회

WHEN 사용자가 공지사항 목록에서 특정 게시글을 클릭하면 THEN 시스템은 해당 게시글의 제목, 내용, 작성일, 조회수를 표시해야 한다.

#### REQ-CS-003 [State-Driven] 공지사항 상단 고정

IF 관리자가 특정 공지사항을 '상단 고정'으로 설정하면 THEN 해당 게시글은 항상 목록 최상단에 노출되어야 한다.

#### REQ-CS-004 [Event-Driven] 공지사항 페이지네이션

WHEN 공지사항 목록의 게시글 수가 페이지당 표시 개수(10건)를 초과하면 THEN 시스템은 페이지네이션을 제공해야 한다.

#### REQ-CS-005 [Ubiquitous] FAQ 카테고리 분류

시스템은 항상 FAQ를 인쇄 업종 특화 카테고리(주문방법/배송/결제/인쇄품질/파일업로드/기타)별로 분류하여 제공해야 한다.

#### REQ-CS-006 [Event-Driven] FAQ 아코디언 토글

WHEN 사용자가 FAQ 항목의 질문을 클릭하면 THEN 해당 답변이 아코디언 방식으로 펼쳐져야 한다.

#### REQ-CS-007 [Event-Driven] FAQ 카테고리 필터링

WHEN 사용자가 특정 카테고리를 선택하면 THEN 해당 카테고리의 FAQ만 필터링되어 표시되어야 한다.

#### REQ-CS-008 [Event-Driven] 비회원 주문조회 요청

WHEN 비회원이 주문번호, 이메일, 휴대폰번호를 입력하고 조회 버튼을 클릭하면 THEN 시스템은 일치하는 주문 정보(주문 상태, 상품 목록, 배송 추적)를 표시해야 한다.

#### REQ-CS-009 [Event-Driven] 비회원 주문조회 실패

WHEN 비회원이 입력한 정보와 일치하는 주문이 없으면 THEN 시스템은 "일치하는 주문을 찾을 수 없습니다" 오류 메시지를 표시하고 회원가입 유도 CTA를 제공해야 한다.

#### REQ-CS-010 [Unwanted] 비회원 주문조회 무제한 시도 금지

시스템은 비회원 주문조회 시도를 무제한으로 허용하지 않아야 한다. (IP당 10회/시간 제한)

#### REQ-CS-011 [State-Driven] 비회원 주문조회 인증 조합

IF 비회원 주문 시 이메일과 휴대폰번호를 모두 수집한 경우 THEN 조회 시 주문번호+이메일+휴대폰번호 3중 조합으로 인증해야 한다.

### 3.2 모듈 2: 관리자 게시판 관리 (Admin Board Management) - 5개 기능

#### REQ-CS-020 [Ubiquitous] 공지사항 관리 CRUD

시스템은 항상 관리자에게 공지사항의 등록, 수정, 삭제, 상단 고정 설정 기능을 제공해야 한다.

#### REQ-CS-021 [Event-Driven] 공지사항 등록

WHEN 관리자가 제목, 내용, 카테고리를 입력하고 등록 버튼을 클릭하면 THEN 시스템은 공지사항을 저장하고 쇼핑몰 공지사항 목록에 즉시 반영해야 한다.

#### REQ-CS-022 [Event-Driven] 공지사항 삭제

WHEN 관리자가 공지사항 삭제를 요청하면 THEN 시스템은 삭제 확인 다이얼로그를 표시하고, 확인 후 해당 게시글을 삭제해야 한다.

#### REQ-CS-023 [Ubiquitous] FAQ 관리 CRUD

시스템은 항상 관리자에게 FAQ의 등록, 수정, 삭제, 카테고리 관리, 노출 순서 변경 기능을 제공해야 한다.

#### REQ-CS-024 [Event-Driven] FAQ 순서 변경

WHEN 관리자가 FAQ 항목의 노출 순서를 드래그앤드롭으로 변경하면 THEN 시스템은 변경된 순서를 저장하고 쇼핑몰에 즉시 반영해야 한다.

#### REQ-CS-025 [Ubiquitous] 1:1문의 관리 기본 원칙

시스템은 항상 관리자에게 1:1문의의 확인, 답변 작성, 상태 관리(대기/처리중/완료) 기능을 제공해야 한다.

#### REQ-CS-026 [Event-Driven] 1:1문의 답변 작성

WHEN 관리자가 1:1문의에 답변을 작성하고 등록하면 THEN 시스템은 답변을 저장하고, 문의 상태를 '완료'로 변경하며, 고객에게 답변 알림(이메일/알림톡)을 발송해야 한다.

#### REQ-CS-027 [Event-Driven] 1:1문의 필터링

WHEN 관리자가 문의 상태(대기/처리중/완료)별 필터를 선택하면 THEN 해당 상태의 문의만 필터링되어 표시되어야 한다.

#### REQ-CS-028 [Ubiquitous] 이용후기 관리 기본 원칙

시스템은 항상 관리자에게 이용후기의 조회, 삭제, 답변 작성, 적립금 지급 관리 기능을 제공해야 한다.

#### REQ-CS-029 [Event-Driven] 이용후기 관리자 임의등록

WHEN 관리자가 이용후기 관리 화면에서 '후기 등록' 버튼을 클릭하고 내용을 작성하면 THEN 시스템은 관리자 작성 후기를 등록하고 쇼핑몰에 노출해야 한다.

#### REQ-CS-030 [Event-Driven] 이용후기 삭제 시 적립금 회수

WHEN 관리자가 적립금이 지급된 이용후기를 삭제하면 THEN 시스템은 해당 회원의 적립금을 자동으로 차감해야 한다.

#### REQ-CS-031 [Ubiquitous] 체험단관리 기본 원칙

시스템은 항상 관리자에게 체험단 모집 공고 등록, 신청자 관리, 당첨자 선정, 체험 후기 관리 기능을 제공해야 한다.

#### REQ-CS-032 [Event-Driven] 체험단 모집 공고 등록

WHEN 관리자가 체험단 모집 공고(제목, 내용, 모집기간, 모집인원, 체험상품)를 입력하고 등록하면 THEN 시스템은 공고를 저장하고 상태를 '모집중'으로 설정해야 한다.

#### REQ-CS-033 [Event-Driven] 체험단 당첨자 선정

WHEN 관리자가 신청자 목록에서 당첨자를 선택하고 '당첨 처리' 버튼을 클릭하면 THEN 시스템은 당첨자 상태를 변경하고 당첨 알림을 발송해야 한다.

#### REQ-CS-034 [State-Driven] 체험단 모집 기간 자동 마감

IF 체험단 모집 기간이 종료되면 THEN 시스템은 해당 공고 상태를 '모집마감'으로 자동 변경하고 추가 신청을 차단해야 한다.

#### REQ-CS-035 [Unwanted] 체험단 중복 신청 금지

시스템은 동일 회원이 동일 체험단에 중복 신청하는 것을 허용하지 않아야 한다.

### 3.3 모듈 3: 관리자 등록/관리 (Admin Management) - 1개 기능

#### REQ-CS-040 [Ubiquitous] 관리자 역할 체계

시스템은 항상 관리자 역할을 3단계(대표관리자/운영자/부운영자)로 구분하고, 각 역할에 대한 권한 그룹을 관리해야 한다.

#### REQ-CS-041 [Event-Driven] 관리자 계정 등록

WHEN 대표관리자가 새 운영자 정보(이름, 이메일, 역할, 권한 그룹)를 입력하고 등록하면 THEN 시스템은 운영자 계정을 생성하고 초기 비밀번호를 이메일로 발송해야 한다.

#### REQ-CS-042 [Event-Driven] 관리자 권한 변경

WHEN 대표관리자가 특정 운영자의 역할 또는 권한 그룹을 변경하면 THEN 시스템은 변경 사항을 즉시 반영하고 해당 운영자의 다음 로그인부터 적용해야 한다.

#### REQ-CS-043 [Event-Driven] 관리자 계정 비활성화

WHEN 대표관리자가 특정 운영자 계정을 비활성화하면 THEN 시스템은 해당 계정의 로그인을 차단하고 관리자 목록에서 비활성 상태로 표시해야 한다.

#### REQ-CS-044 [Unwanted] 대표관리자 단독 변경 금지

시스템은 유일한 대표관리자 계정을 삭제하거나 비활성화하는 것을 허용하지 않아야 한다.

#### REQ-CS-045 [State-Driven] 관리자 권한별 메뉴 접근 제어

IF 운영자의 권한 그룹에 특정 관리 메뉴 접근 권한이 없으면 THEN 해당 메뉴는 사이드바에서 비표시 처리해야 한다.

#### REQ-CS-046 [Optional] 2FA 인증 도입

가능하면 관리자 로그인 시 2FA(이메일 OTP) 인증 기능을 제공해야 한다. (P2 이후 검토)

## 4. 비기능 요구사항

### 4.1 성능

- 공지사항/FAQ 목록 로딩: 1초 이내 (캐시 활용)
- 비회원 주문조회 응답: 2초 이내
- 관리자 게시판 CRUD 응답: 500ms 이내

### 4.2 보안

- 비회원 주문조회: Rate Limiting (IP당 10회/시간)
- 관리자 권한: 역할 기반 접근 제어 (RBAC)
- 관리자 세션: 30분 비활동 시 자동 로그아웃
- 체험단 DB: SQL Injection 방지 (Parameterized Query 필수)

### 4.3 접근성

- 게시판 컨텐츠: 스크린리더 호환 (ARIA 라벨 적용)
- FAQ 아코디언: 키보드 네비게이션 지원 (Enter/Space 토글)

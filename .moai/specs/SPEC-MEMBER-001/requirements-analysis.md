## 1. 핵심 의사결정 사항

### KD-01: 로그인 식별자 방식 (A-1-1)

**결정 항목**: 이메일 로그인 vs 아이디 로그인 vs 둘 다

**선택 가능 옵션**:
| 옵션 | 내용 | 장점 | 단점 |
|------|------|------|------|
| A | 이메일 | 기억하기 쉬움, 글로벌 표준 | 기존 아이디 회원 마이그레이션 필요 |
| B | 아이디 | 기존 방식 유지 | 아이디 분실 위험, 고객지원 부담 |
| C | 둘 다 | 유연성 최대 | UI 복잡, 정책 일관성 저하 |

**경쟁사 벤치마크**:
- 레드프린팅(redprinting): 이메일 방식
- 와우프레스(wowpress): 아이디 방식

**권장 결정**: **이메일 방식 (옵션 A)**

**근거**:
- 인쇄 업계 B2B/B2C 고객은 이메일을 업무 기반으로 사용하여 기억 부담 감소
- 이메일은 아이디 찾기 불필요 (아이디 = 이메일이므로 자가 복구 가능)
- 향후 SNS 연동 시 이메일 기준 계정 통합이 자연스러움
- shopby에서 이메일 로그인 설정 시 기존 아이디 회원은 마이그레이션 계획 수립 필요

**shopby 구현 가능 여부**: 가능 (관리자 > 기본설정에서 전환)

### KD-02: 아이디 찾기 방법 및 노출 방식 (A-1-2)

**결정 항목**: 찾기 방법(휴대전화/이메일 인증), 결과 마스킹 여부

**선택 가능 옵션**:
| 항목 | 옵션 |
|------|------|
| 인증 방법 | 이름+휴대전화 조합 (shopby API 기본 제공) |
| 결과 노출 | 전체 노출 / 일부 마스킹 / 마스킹+이메일 발송 |

**권장 결정**: **일부 마스킹 + 전체 이메일 발송 옵션 제공**

**근거**:
- 마스킹: 개인정보 보호 및 계정 탈취 방지 (예: ab***@naver.com)
- 이메일 전체 발송: 고객이 원할 경우 보안 채널로 확인 가능
- shopby API는 이름+휴대전화 조합만 지원 (추가 인증 방식 미제공)
- KD-01에서 이메일 방식 선택 시 "아이디 찾기" 화면은 "가입 이메일 찾기"로 명칭 변경 필요

**shopby 구현 가능 여부**: 마스킹은 프론트엔드 처리, API는 전체 이메일 반환

---

### KD-03: 비밀번호 찾기 방식 (A-1-3)

**결정 항목**: 이메일 재설정 링크 vs 임시 비밀번호 발송

**선택 가능 옵션**:
| 옵션 | 내용 | 보안성 |
|------|------|--------|
| A | 이메일 재설정 링크 (시간 제한) | 높음 |
| B | 임시 비밀번호 SMS 발송 | 낮음 |
| C | 이메일 재설정 링크 + SMS 알림 | 높음 |

**권장 결정**: **이메일 재설정 링크 방식 (옵션 A)**

**근거**:
- OWASP 권장 방식: 시간 제한 토큰 기반 재설정 링크
- 임시 비밀번호는 평문 전송 취약점 존재 (보안 취약)
- SMS 발송 추가 비용 (건당 8~15원) 절감
- shopby 관리자에서 이메일 템플릿 커스텀 가능
- 알림톡 도입 시 별도 카카오 비즈니스 채널 계약 필요 (P2 검토)

**shopby 구현 가능 여부**: 가능 (shopby 관리자 > 자동메일설정)

### KD-04: 약관동의 14세 확인 사용 여부 (A-2-1)

**결정 항목**: 14세 미만 이용 제한 확인 체크박스 사용 여부

**선택 가능 옵션**:
| 옵션 | 내용 | 법적 의미 |
|------|------|----------|
| A | 14세 이상 확인 체크박스 사용 | 정보통신망법 준수 강화 |
| B | 미사용 | UI 간소화 |

**권장 결정**: **14세 이상 확인 체크박스 사용 (옵션 A)**

**근거**:
- 정보통신망 이용촉진 및 정보보호 등에 관한 법률 제31조: 14세 미만 아동의 개인정보 수집 시 법정대리인 동의 필요
- 인쇄물 B2C 서비스에서 14세 미만 이용자가 소수이나, 법적 리스크 최소화 필요
- shopby 관리자에서 사용/미사용 전환 가능

**shopby 구현 가능 여부**: 가능 (관리자 > 가입설정)

### KD-05: 회원가입 필수/선택 필드 구성 (A-2-2)

**결정 항목**: 가입 시 수집 정보 범위

**권장 결정**:

**필수 필드**:
| 필드 | 이유 |
|------|------|
| 이메일(로그인ID) | 계정 식별자 |
| 비밀번호 | 인증 수단 |
| 이름 | 주문/배송 처리 |
| 휴대전화 | 본인인증, 주문 알림 |

**선택 필드** (가입 후 마이페이지에서 입력 가능):
| 필드 | 이유 |
|------|------|
| 닉네임 | 리뷰 표시용 (미입력 시 이름으로 대체) |
| 생년월일 | 생일 쿠폰 발송 (마케팅 목적) |
| 사업자번호 | B2B 세금계산서 발급 (법인 고객) |

**제외 필드**:
| 필드 | 제외 이유 |
|------|----------|
| 성별 | 인쇄물 도메인에서 마케팅 활용도 낮음 |
| 주소 | 첫 주문 시 입력 (가입 장벽 최소화) |
| 추천인 코드 | 사용 정책 미정 (P2 검토) |

**근거**: 가입 장벽 최소화로 전환율 향상, 인쇄 도메인에서 불필요한 정보 수집 지양

### KD-06: 휴대전화 인증 방식 (A-2-4)

**결정 항목**: SMS 인증번호 vs PASS 본인인증

**선택 가능 옵션**:
| 옵션 | 비용 | 구현 난이도 | 1인1계정 효과 |
|------|------|------------|--------------|
| A | SMS 인증번호 | 건당 8~15원 | 낮음 (번호 도용 가능) | 낮음 |
| B | PASS 본인인증 | 별도 계약 필요 | 높음 | 높음 (CI 기반) |
| C | SMS + PASS 선택 | 혼합 | 높음 | 중간 |

**권장 결정**: **SMS 인증번호 방식 (옵션 A) + 1인1계정 제한 미사용**

**근거**:
- 인쇄물 주문 특성상 B2B 고객은 법인 폰으로 가입 가능 (PASS 제한 우회)
- PASS 본인인증은 별도 KCB/NICE 계약 필요 (초기 비용 발생)
- SMS 인증으로도 스팸 회원 방지 충분
- 1인1계정 제한 미사용 시: B2B 고객이 개인/법인 계정을 분리 사용 가능
- PASS 도입은 P2(2차 런칭)에서 검토 권장

**shopby 구현 가능 여부**: SMS 방식 가능 (NHN Cloud 등 SMS 업체 연동 필요)

### KD-07: 비회원 인쇄물 주문 허용 범위 (A-2-GUEST)

**결정 항목**: 비회원 주문 허용 여부 및 허용 범위

**선택 가능 옵션**:
| 옵션 | 내용 |
|------|------|
| A | 전체 허용 (굿즈, 인쇄물, 출력물 모두) |
| B | 일반 상품만 허용 (굿즈 등), 인쇄물 파일업로드 필요 시 회원 필수 |
| C | 비회원 주문 불허 |

**경쟁사 벤치마크**:
- 레드프린팅: 비회원 주문 가능 (주문번호+휴대전화)

**권장 결정**: **옵션 B - 제한적 허용**

**근거**:
- 인쇄물 주문은 파일 업로드, 편집 히스토리, 교정 확인 등 계정 기반 기능 필요
- 비회원은 주문 후 파일 재업로드 요청 처리가 복잡해짐
- 굿즈 등 단순 상품은 비회원 주문 허용하여 구매 장벽 낮춤
- 비회원 주문 시 "회원가입 시 쿠폰 제공" 유도 배너 표시

**허용 범위 정의**:
- 허용: 단순 굿즈 상품 (파일 업로드 불필요)
- 불허: 인쇄/제본 상품 (파일 업로드 필수), 출력물 상품

**shopby 구현 가능 여부**: 비회원 주문 허용 설정 가능, 상품별 제한은 프론트엔드 처리 필요

### KD-08: 회원등급 단계 및 조건 (B-6-GRADE)

**결정 항목**: 등급 단계 수, 산정 기준, 혜택 체계

**경쟁사 벤치마크**:
- 와우프레스: 6단계 (3개월 접수액 기준)

**권장 결정**: **4단계 등급 체계**

| 등급 | 이름 | 기준 (3개월 구매액) | 적립률 | 혜택 |
|------|------|-------------------|--------|------|
| 1 | 일반 | 가입 즉시 | 1% | - |
| 2 | 실버 | 10만원 이상 | 2% | 월 5,000원 쿠폰 |
| 3 | 골드 | 30만원 이상 | 3% | 월 10,000원 쿠폰 |
| 4 | VIP | 50만원 이상 | 5% | 월 30,000원 쿠폰 + 무료배송 |

**근거**:
- 4단계: 관리 복잡성과 고객 인지도 사이의 최적 균형
- 산정 기간 3개월: 계절성을 반영하면서 자주 등급 변동 방지
- 구매금액 기준: 인쇄물은 단가가 높아 건수보다 금액 기준이 적합
- VIP 기준 50만원: 후니프린팅 인쇄물 평균 객단가 고려

**shopby 구현 가능 여부**: 완전 가능 (관리자 > 회원등급 설정)

## 2. 모듈 1: 인증 (Authentication)

### 2.1 기능 목록

| 기능 | Policy ID | shopby_class | 우선순위 |
|------|-----------|-------------|---------|
| 일반 로그인 | A-1-1 | NATIVE | P1 |

| 비밀번호 찾기 | A-1-3 | NATIVE | P1 |

| SNS 로그인 (구글/애플) | A-1-SNS2 | EXTERNAL | P2 |

### 2.2 EARS 요구사항

#### 2.2.1 일반 로그인 (A-1-1)

**Ubiquitous (항상 적용)**

- 시스템은 항상 이메일+비밀번호 조합으로 사용자를 인증해야 한다.
- 시스템은 항상 JWT accessToken(30분) + refreshToken을 발급해야 한다.
- 시스템은 항상 비밀번호를 평문으로 저장하지 않아야 한다.
- 시스템은 항상 HTTPS를 통해 인증 데이터를 전송해야 한다.

**Event-Driven (이벤트 기반)**

- WHEN 사용자가 올바른 이메일과 비밀번호를 입력하고 로그인 버튼을 클릭하면 THEN 시스템은 accessToken과 refreshToken을 발급하고 메인 페이지로 이동해야 한다.
- WHEN 사용자가 잘못된 이메일 또는 비밀번호를 입력하면 THEN 시스템은 "이메일 또는 비밀번호가 일치하지 않습니다" 오류 메시지를 표시해야 한다.
- WHEN 로그인 실패가 5회 누적되면 THEN 시스템은 해당 계정을 일시적으로 잠금 처리하고 잠금 시간을 안내해야 한다.
- WHEN 자동 로그인 체크박스를 선택하고 로그인하면 THEN 시스템은 refreshToken을 로컬스토리지에 저장하여 세션 만료 후에도 자동 갱신해야 한다.
- WHEN accessToken이 만료되면 THEN 시스템은 refreshToken으로 자동으로 accessToken을 갱신해야 한다.
- WHEN 로그아웃 버튼을 클릭하면 THEN 시스템은 토큰을 삭제하고 로그인 페이지로 이동해야 한다.

**State-Driven (상태 기반)**

- IF 계정이 잠금 상태이면 THEN 시스템은 로그인을 거부하고 잠금 해제 시간을 표시해야 한다.
- IF 로그인된 상태에서 로그인 페이지에 접근하면 THEN 시스템은 메인 페이지로 리다이렉트해야 한다.

**Unwanted (금지 사항)**

- 시스템은 로그인 실패 시 이메일 존재 여부를 노출하지 않아야 한다 (계정 열거 공격 방지).
- 시스템은 비밀번호를 URL 파라미터로 전달하지 않아야 한다.
- 시스템은 accessToken을 쿠키에 httpOnly 없이 저장하지 않아야 한다.

#### 2.2.2 아이디 찾기 (A-1-2)

**Event-Driven (이벤트 기반)**

- WHEN 사용자가 이름과 휴대전화번호를 입력하고 확인 버튼을 클릭하면 THEN 시스템은 일치하는 이메일을 조회하고 마스킹하여 표시해야 한다 (예: ab***@naver.com).
- WHEN 마스킹된 이메일이 표시된 상태에서 "이메일 전체 발송" 버튼을 클릭하면 THEN 시스템은 해당 이메일로 전체 이메일 주소를 발송해야 한다.
- WHEN 입력한 이름+휴대전화 조합에 해당하는 계정이 없으면 THEN 시스템은 "일치하는 계정을 찾을 수 없습니다" 메시지를 표시해야 한다.

**Unwanted (금지 사항)**

- 시스템은 존재하지 않는 계정 조회 시도를 반복할 경우(5회 이상) 일시적으로 해당 IP의 요청을 차단해야 한다.

#### 2.2.3 비밀번호 찾기 (A-1-3)

**Event-Driven (이벤트 기반)**

- WHEN 사용자가 이메일, 이름, 휴대전화를 입력하고 재설정 링크 발송 버튼을 클릭하면 THEN 시스템은 해당 이메일로 비밀번호 재설정 링크를 발송해야 한다.
- WHEN 비밀번호 재설정 링크를 클릭하면 THEN 시스템은 새 비밀번호 입력 화면을 표시해야 한다.
- WHEN 재설정 링크 발송 후 1분 이내에 재발송 버튼을 클릭하면 THEN 시스템은 "1분 후에 다시 시도해주세요" 메시지를 표시하고 발송을 차단해야 한다.

**State-Driven (상태 기반)**

- IF 비밀번호 재설정 링크가 24시간 경과하면 THEN 시스템은 해당 링크를 만료 처리하고 재발송을 안내해야 한다.
- IF 재설정 링크가 이미 사용된 경우 THEN 시스템은 "이미 사용된 링크입니다" 오류를 표시해야 한다.

#### 2.2.4 SNS 로그인 카카오/네이버 (A-1-SNS)

**Event-Driven (이벤트 기반)**

- WHEN 카카오 로그인 버튼을 클릭하면 THEN 시스템은 카카오 OAuth 인증 페이지로 이동해야 한다.
- WHEN 카카오 인증 완료 후 콜백을 받으면 THEN 시스템은 기존 계정 연동 여부를 확인하고 자동 로그인 처리해야 한다.
- WHEN SNS 최초 로그인이고 필수 정보(이름, 휴대전화)가 없으면 THEN 시스템은 추가 정보 입력 화면을 표시해야 한다.
- WHEN SNS 이메일이 기존 일반 계정과 동일하면 THEN 시스템은 "이미 가입된 이메일입니다. 기존 계정과 연동할까요?" 안내를 표시해야 한다.
- WHEN 네이버 로그인 버튼을 클릭하면 THEN 시스템은 네이버 OAuth 인증 페이지로 이동해야 한다.

**Ubiquitous (항상 적용)**

- 시스템은 항상 카카오 공식 디자인 가이드(노란색 배경)를 준수한 버튼을 사용해야 한다.
- 시스템은 항상 네이버 공식 디자인 가이드(초록색 배경)를 준수한 버튼을 사용해야 한다.

**Optional (조건부)**

- WHERE 구글/애플 로그인이 P2에서 도입될 경우 시스템은 해당 플랫폼의 공식 OAuth 2.0 흐름을 구현해야 한다.

### 2.3 인증 모듈 인수 기준

#### 시나리오 1: 정상 이메일 로그인

```
GIVEN 가입된 이메일 계정이 있고
WHEN 올바른 이메일과 비밀번호로 로그인 시도하면
THEN accessToken과 refreshToken이 발급되고
AND 메인 페이지로 이동하며
AND "로그인되었습니다" 알림이 표시된다
```

```
GIVEN 가입된 계정이 있고
WHEN 잘못된 비밀번호로 5회 연속 로그인 시도하면
THEN 계정이 일시적으로 잠금 처리되고
AND "로그인 시도 횟수를 초과했습니다. 30분 후 다시 시도해주세요" 메시지가 표시되고

```

```
WHEN 카카오 로그인을 완료하면
```

```
GIVEN 비밀번호 찾기 요청으로 재설정 링크가 발송된 상태에서
WHEN 24시간 경과 후 재설정 링크를 클릭하면
THEN "링크가 만료되었습니다" 오류가 표시되고
AND 재설정 이메일 재발송 버튼이 제공된다
```

## 3. 모듈 2: 회원가입 (Registration)

### 3.1 기능 목록

| 기능 | Policy ID | shopby_class | 우선순위 |
|------|-----------|-------------|---------|
| 약관동의 | A-2-1 | SKIN | P1 |
| 정보입력 | A-2-2 | SKIN | P1 |
| 이메일 중복확인 | A-2-3 | NATIVE | P1 |
| 휴대전화 인증 | A-2-4 | NATIVE | P1 |
| 가입완료 메일 발송 | A-2-5 | NATIVE | P1 |

### 3.2 EARS 요구사항

#### 3.2.1 약관동의 (A-2-1)

**Ubiquitous (항상 적용)**

- 시스템은 항상 가입 절차의 첫 단계에서 약관동의 화면을 표시해야 한다.
- 시스템은 항상 이용약관과 개인정보처리방침을 필수 약관으로 표시해야 한다.
- 시스템은 항상 마케팅 정보 수신동의를 선택 약관으로 표시해야 한다.
- 시스템은 항상 14세 이상 확인 체크박스를 표시해야 한다 (KD-04 결정).

**Event-Driven (이벤트 기반)**

- WHEN "전체 동의" 체크박스를 클릭하면 THEN 시스템은 모든 약관(필수+선택+14세 확인)을 일괄 선택/해제해야 한다.
- WHEN 약관 제목 영역의 "보기" 버튼을 클릭하면 THEN 시스템은 해당 약관 내용을 아코디언으로 펼쳐야 한다.
- WHEN 모든 필수 약관에 동의하면 THEN 시스템은 "다음" 버튼을 활성화해야 한다.

**Unwanted (금지 사항)**

- 시스템은 필수 약관에 미동의한 상태에서 다음 단계 진행을 허용하지 않아야 한다.
- 시스템은 약관 내용 없이 약관 제목만 표시하지 않아야 한다.

#### 3.2.2 정보입력 (A-2-2)

**Ubiquitous (항상 적용)**

- 시스템은 항상 이메일, 비밀번호, 비밀번호 확인, 이름, 휴대전화를 필수 입력 항목으로 요구해야 한다.
- 시스템은 항상 비밀번호를 영문, 숫자, 특수문자 조합 8자 이상으로 검증해야 한다.
- 시스템은 항상 이메일 형식을 RFC 5322 표준으로 검증해야 한다.

**Event-Driven (이벤트 기반)**

- WHEN 이메일 입력 필드에 300ms 이상 입력이 없으면 THEN 시스템은 이메일 중복 확인을 자동으로 실행해야 한다 (debounce).
- WHEN 비밀번호를 입력하면 THEN 시스템은 비밀번호 강도를 실시간으로 "약함/보통/강함"으로 표시해야 한다.
- WHEN "가입하기" 버튼을 클릭하면 THEN 시스템은 모든 필수 필드 검증 후 회원 가입 API를 호출해야 한다.
- WHEN 회원 가입이 성공하면 THEN 시스템은 가입 완료 페이지를 표시하고 shopby 자동메일을 트리거해야 한다.

**State-Driven (상태 기반)**

- IF 이메일 중복 확인이 완료되지 않은 경우 THEN 시스템은 "가입하기" 버튼을 비활성화해야 한다.
- IF 휴대전화 인증이 완료되지 않은 경우 THEN 시스템은 "가입하기" 버튼을 비활성화해야 한다.

#### 3.2.3 이메일 중복확인 (A-2-3)

**Event-Driven (이벤트 기반)**

- WHEN 이메일 형식이 올바르고 debounce 시간이 경과하면 THEN 시스템은 `/members/id/exist` API를 호출하여 중복 여부를 확인해야 한다.
- WHEN 이메일이 사용 가능하면 THEN 시스템은 초록색 체크 아이콘과 "사용 가능한 이메일입니다" 메시지를 표시해야 한다.
- WHEN 이메일이 이미 사용 중이면 THEN 시스템은 빨간색 X 아이콘과 "이미 사용 중인 이메일입니다" 메시지를 표시해야 한다.

**Ubiquitous (항상 적용)**

- 시스템은 항상 이메일 형식 검증 후 API를 호출해야 한다 (불필요한 API 호출 최소화).
- 시스템은 항상 API 호출 빈도를 초당 10회 이내로 제한해야 한다 (shopby API 정책).

#### 3.2.4 휴대전화 인증 (A-2-4)

**Event-Driven (이벤트 기반)**

- WHEN "인증번호 발송" 버튼을 클릭하면 THEN 시스템은 해당 휴대전화로 6자리 인증번호 SMS를 발송하고 3분 타이머를 시작해야 한다.
- WHEN 올바른 인증번호를 입력하면 THEN 시스템은 "인증이 완료되었습니다" 메시지를 표시하고 휴대전화 필드를 비활성화해야 한다.
- WHEN 잘못된 인증번호를 입력하면 THEN 시스템은 "인증번호가 일치하지 않습니다" 오류를 표시해야 한다.
- WHEN 3분 타이머가 만료되면 THEN 시스템은 "인증번호가 만료되었습니다. 재발송 해주세요" 메시지를 표시해야 한다.
- WHEN 인증번호 발송 후 1분 이내 재발송 시도하면 THEN 시스템은 재발송을 차단하고 남은 시간을 안내해야 한다.

**Unwanted (금지 사항)**

- 시스템은 3분 만료 후에도 이전 인증번호로 인증을 허용하지 않아야 한다.

#### 3.2.5 가입완료 메일 발송 (A-2-5)

**Event-Driven (이벤트 기반)**

- WHEN 회원 가입 API(`/members` POST)가 성공하면 THEN shopby는 자동으로 가입 완료 이메일을 발송해야 한다.
- WHEN 가입 완료 화면이 표시되면 THEN 시스템은 신규 가입 혜택(쿠폰 10,000원, 정책 데이터 #12 참조)을 안내해야 한다.

**Optional (조건부)**

- WHERE 신규 가입 쿠폰이 설정된 경우 시스템은 가입 완료 화면에 쿠폰 금액을 표시해야 한다.

### 3.3 회원가입 모듈 인수 기준

#### 시나리오 1: 정상 회원가입 플로우

```
GIVEN 비회원 상태에서
WHEN 이용약관 및 개인정보처리방침에 동의하고
AND 이메일, 비밀번호, 이름, 휴대전화를 입력하고
AND 이메일 중복 확인이 "사용 가능"이고
AND 휴대전화 SMS 인증을 완료하고
AND "가입하기" 버튼을 클릭하면
THEN 회원 계정이 생성되고
AND 가입 완료 이메일이 발송되고
AND 신규 가입 쿠폰(10,000원)이 발급되고
AND 가입 완료 페이지가 표시된다
```

#### 시나리오 2: 이미 가입된 이메일로 재가입 시도

```
GIVEN 이미 이메일로 가입된 계정이 있을 때
WHEN 동일 이메일로 이메일 중복 확인을 실행하면
THEN "이미 사용 중인 이메일입니다" 오류가 표시되고
AND "가입하기" 버튼이 비활성화 상태를 유지한다
```

#### 시나리오 3: 필수 약관 미동의 시 다음 단계 차단

```
GIVEN 약관동의 화면에서
WHEN 필수 약관(이용약관, 개인정보처리방침) 중 하나라도 미동의한 상태에서 "다음" 버튼을 클릭하면
THEN 페이지 이동이 차단되고
AND "필수 약관에 동의해주세요" 오류 메시지가 표시된다
```

#### 시나리오 4: SMS 인증 타이머 만료

```
GIVEN 인증번호 발송 후 3분이 경과한 상태에서
WHEN 만료된 인증번호를 입력하면
THEN "인증번호가 만료되었습니다" 오류가 표시되고
AND "인증번호 재발송" 버튼이 활성화된다
```

## 4. 모듈 3: 회원관리 (Member Management)

### 4.1 기능 목록

| 기능 | Policy ID | shopby_class | 우선순위 |
|------|-----------|-------------|---------|
| 비회원 주문 허용 | A-2-GUEST | NATIVE | P1 |
| 회원등급 체계 | B-6-GRADE | NATIVE | P1 |
| 회원관리 (관리자) | B-6-1 | NATIVE | P2 |

### 4.2 EARS 요구사항

#### 4.2.1 비회원 주문 허용 (A-2-GUEST)

**Ubiquitous (항상 적용)**

- 시스템은 항상 비회원 주문 시 이름, 휴대전화, 이메일을 필수 수집해야 한다.
- 시스템은 항상 비회원 주문 조회를 주문번호+휴대전화 조합으로 제공해야 한다.

**Event-Driven (이벤트 기반)**

- WHEN 비회원이 단순 굿즈 상품을 주문하려 할 때 THEN 시스템은 비회원 정보 입력 화면을 제공해야 한다.
- WHEN 비회원이 인쇄물/제본 상품(파일 업로드 필요)을 주문하려 할 때 THEN 시스템은 "인쇄물 주문은 회원만 가능합니다. 가입 시 혜택을 받아보세요" 안내와 함께 로그인/가입 화면으로 유도해야 한다.
- WHEN 비회원 주문이 완료되면 THEN 시스템은 주문번호를 이메일 및 SMS로 발송해야 한다.

**State-Driven (상태 기반)**

- IF 비회원 주문 상태에서 쿠폰 사용을 시도하면 THEN 시스템은 "쿠폰은 회원만 사용 가능합니다" 메시지를 표시해야 한다.
- IF 비회원 주문 상태에서 프린팅머니 사용을 시도하면 THEN 시스템은 "프린팅머니는 회원만 사용 가능합니다" 메시지를 표시해야 한다.

**Unwanted (금지 사항)**

- 시스템은 비회원 주문 정보를 90일 초과하여 보관하지 않아야 한다 (shopby 기본 정책).

#### 4.2.2 회원등급 체계 (B-6-GRADE)

**Ubiquitous (항상 적용)**

- 시스템은 항상 4단계 등급 체계(일반/실버/골드/VIP)를 유지해야 한다 (KD-08).
- 시스템은 항상 3개월 구매금액을 기준으로 등급을 산정해야 한다.
- 시스템은 항상 등급 산정 주기(shopby 설정)에 따라 자동으로 등급을 재산정해야 한다.

**Event-Driven (이벤트 기반)**

- WHEN 회원이 마이페이지 상단을 방문하면 THEN 시스템은 현재 등급 배지와 "다음 등급까지 N원 남았습니다" 프로그레스 바를 표시해야 한다.
- WHEN 등급이 상승하면 THEN 시스템은 등급 상승 알림 이메일 또는 SMS를 발송해야 한다.
- WHEN 등급별 혜택 페이지를 방문하면 THEN 시스템은 전체 등급 조건 및 혜택 비교표를 표시해야 한다.

**State-Driven (상태 기반)**

- IF VIP 등급 회원이 주문하면 THEN 시스템은 자동으로 무료배송 혜택을 적용해야 한다.

#### 4.2.3 회원관리 관리자 (B-6-1)

**Ubiquitous (항상 적용)**

- 시스템은 항상 관리자에게 회원의 기본 정보, 주문내역, 등급 정보를 제공해야 한다.

**Event-Driven (이벤트 기반)**

- WHEN 관리자가 회원 목록에서 검색하면 THEN 시스템은 이름, 이메일, 휴대전화로 필터링된 결과를 표시해야 한다.
- WHEN 관리자가 특정 회원의 상세 정보를 클릭하면 THEN 시스템은 회원의 전체 주문내역, 쿠폰, 프린팅머니 잔액을 표시해야 한다.

### 4.3 회원관리 모듈 인수 기준

#### 시나리오 1: 비회원 인쇄물 주문 차단

```
GIVEN 비회원이 인쇄물 상품을 장바구니에 담고 주문을 시도할 때
WHEN 주문 버튼을 클릭하면
THEN "인쇄물 주문은 회원만 가능합니다" 안내가 표시되고
AND 로그인/가입 유도 CTA 버튼이 표시되고
AND 가입 시 혜택(신규 쿠폰) 안내가 함께 표시된다
```

#### 시나리오 2: 회원 등급 상승

```
GIVEN 실버 등급 회원이 3개월 내 구매액이 30만원을 초과했을 때
WHEN shopby 등급 산정 배치가 실행되면
THEN 해당 회원의 등급이 골드로 변경되고
AND 등급 상승 알림 이메일이 발송되고
AND 마이페이지에서 변경된 등급이 표시된다
```

#### 시나리오 3: 비회원 주문 조회

```
GIVEN 비회원이 주문을 완료한 상태에서
WHEN 주문번호와 휴대전화번호를 입력하고 주문 조회를 하면
THEN 해당 주문의 상태, 배송 정보, 상품 내역이 표시된다
```

### 5.2 EARS 요구사항

#### 5.2.1 탈퇴회원 관리 (B-6-2)

**Ubiquitous (항상 적용)**

- 시스템은 항상 탈퇴 회원 정보를 전자상거래법에 따라 5년간 보관해야 한다.
- 시스템은 항상 탈퇴 회원의 개인식별정보를 마스킹하여 보관해야 한다.

**Event-Driven (이벤트 기반)**

- WHEN 관리자가 탈퇴 회원 목록을 조회하면 THEN 시스템은 탈퇴일, 탈퇴 사유, 마스킹된 정보를 표시해야 한다.
- WHEN 미완료 주문이 있는 회원이 탈퇴를 요청하면 THEN 시스템은 탈퇴를 거부하고 "진행 중인 주문이 있어 탈퇴가 불가합니다" 메시지를 표시해야 한다.
- WHEN 회원이 탈퇴를 완료하면 THEN 시스템은 해당 회원의 잔여 쿠폰과 프린팅머니를 즉시 소멸시켜야 한다.

**State-Driven (상태 기반)**

- IF 탈퇴 후 재가입 제한 기간(30일 권장) 이내이면 THEN 시스템은 동일 이메일로 재가입을 차단해야 한다.

#### 5.2.2 프린팅머니 관리 (B-6-3)

**프린팅머니**는 후니프린팅의 적립금 명칭 (shopby 적립금 기능 활용)

**Ubiquitous (항상 적용)**

- 시스템은 항상 프린팅머니 지급/차감 내역을 관리자에게 제공해야 한다.
- 시스템은 항상 프린팅머니 사용 시 최소 1원 단위로 처리해야 한다.

**Event-Driven (이벤트 기반)**

- WHEN 관리자가 특정 회원에게 프린팅머니를 지급하면 THEN 시스템은 지급 사유와 금액을 기록해야 한다.
- WHEN 관리자가 프린팅머니를 차감하면 THEN 시스템은 차감 사유와 금액을 기록하고 해당 회원에게 알림을 발송해야 한다.
- WHEN 관리자가 일괄 지급 기능을 사용하면 THEN 시스템은 지정 조건(등급, 구매이력 등)에 해당하는 회원 전체에게 프린팅머니를 지급해야 한다.

**State-Driven (상태 기반)**

- IF 회원이 탈퇴하면 THEN 시스템은 잔여 프린팅머니를 즉시 소멸시켜야 한다.
- IF 프린팅머니 잔액이 주문 금액보다 적으면 THEN 시스템은 차액을 다른 결제 수단으로 처리할 수 있도록 해야 한다.

#### 5.2.3 쿠폰관리 (B-6-4)

**Ubiquitous (항상 적용)**

- 시스템은 항상 쿠폰 발행 시 쿠폰 코드, 할인 금액, 유효기간, 최소 주문금액을 설정할 수 있어야 한다.
- 시스템은 항상 쿠폰의 사용 대상(전체/특정 상품/특정 카테고리)을 설정할 수 있어야 한다.

**Event-Driven (이벤트 기반)**

- WHEN 관리자가 신규 쿠폰을 생성하면 THEN 시스템은 쿠폰 코드를 자동 생성하고 유효성을 검증해야 한다.
- WHEN 관리자가 쿠폰을 특정 회원에게 발급하면 THEN 시스템은 해당 회원의 쿠폰 지갑에 추가해야 한다.
- WHEN 쿠폰 유효기간이 만료되면 THEN 시스템은 해당 쿠폰을 자동으로 비활성화해야 한다.
- WHEN 신규 회원이 가입 완료하면 THEN 시스템은 신규 가입 쿠폰(10,000원, 최소 주문금액 50,000원)을 자동 발급해야 한다 (정책 #12, #13).

**Optional (조건부)**

- WHERE 출력상품 전용 쿠폰 설정이 가능할 경우 시스템은 인쇄/제본 카테고리에만 적용되는 쿠폰을 발행할 수 있어야 한다.

### 5.3 관리자 회원운영 모듈 인수 기준

#### 시나리오 1: 신규 가입 쿠폰 자동 발급

```
GIVEN 새로운 회원이 가입을 완료했을 때
WHEN 가입 완료 이벤트가 발생하면
THEN 10,000원 쿠폰이 자동으로 발급되고 (최소 주문금액 50,000원)
AND 쿠폰 유효기간이 30일로 설정되고
AND 가입 완료 이메일에 쿠폰 정보가 포함된다
```

#### 시나리오 2: 미완료 주문 있는 회원 탈퇴 거부

```
GIVEN 배송 중인 주문이 있는 회원이 탈퇴를 요청했을 때
WHEN 탈퇴 요청을 제출하면
THEN "진행 중인 주문 N건이 있어 탈퇴가 불가합니다" 오류가 표시되고
AND 주문 조회 링크가 제공된다
```

#### 시나리오 3: 관리자 프린팅머니 일괄 지급

```
GIVEN 관리자가 VIP 등급 회원 전체에게 프린팅머니 지급을 실행했을 때
WHEN 일괄 지급을 확정하면
THEN VIP 등급 회원 전체에게 지정된 금액이 지급되고
AND 각 회원에게 지급 알림이 발송되고
AND 관리자 화면에 지급 완료 내역이 기록된다
```

## 6. 엣지 케이스 및 리스크

### 6.1 인증 관련 엣지 케이스

| 케이스 | 발생 조건 | 처리 방법 |
|--------|----------|----------|
| SNS 이메일 없음 | 카카오/네이버 이메일 미동의 | 이메일 직접 입력 화면 제공 |
| accessToken 갱신 실패 | refreshToken도 만료 | 로그인 페이지로 강제 이동 |
| 동시 로그인 | 다른 기기에서 로그인 | 기존 세션 유지 또는 새 세션 우선 (shopby 정책 확인 필요) |
| 비밀번호 재설정 링크 재사용 | 이미 사용된 링크 재클릭 | "이미 사용된 링크" 오류 표시 |
| 이메일 변경 불가 정책 | 이메일을 변경하려는 요청 | "이메일은 변경이 불가합니다" 안내 |

### 6.2 회원가입 관련 엣지 케이스

| 케이스 | 발생 조건 | 처리 방법 |
|--------|----------|----------|
| SMS 미수신 | 통신 장애 또는 번호 오류 | 1분 후 재발송, 3회 실패 시 고객센터 안내 |
| 가입 도중 이탈 | 정보 입력 중 브라우저 닫기 | 입력된 데이터 보존 없음 (개인정보 보호) |
| 동일 휴대전화 다중 가입 | 1인1계정 미설정 시 | 허용 (법인 폰으로 여러 계정 가능) |
| SNS 가입 후 일반 가입 | 동일 이메일 사용 | "이미 SNS로 가입된 이메일" 안내 |

### 6.3 주요 리스크

| 리스크 | 심각도 | 대응 방안 |
|--------|--------|----------|
| shopby API 다운 시 로그인 불가 | 높음 | 유지보수 공지, fallback 화면 |
| SMS 인증 비용 급증 (가입 봇) | 중간 | IP별 인증 발송 횟수 제한 |
| 이메일 전환 시 기존 회원 혼란 | 높음 | 마이그레이션 안내 이메일 + 지원 기간 |
| 회원탈퇴 시 개인정보 미삭제 | 높음 | shopby 기본 5년 보관 정책 확인 필수 |
| PASS 미도입으로 소셜로그인 계정 탈취 | 중간 | 2FA 선택 제공 (P2) |

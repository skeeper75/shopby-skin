# SPEC-PLAN-001 실행 계획

## 1. 실행 계획 개요

SPEC-PLAN-001은 **메타 SPEC(기획 문서)**으로, 코드 구현이 아닌 **기획 결정과 문서 산출물**이 목표이다. 본 계획은 25개 운영정책 결정, 88개 기능 로드맵 확정, 9개 CUSTOM 모듈 정의, 7개 후속 SPEC 계획 수립의 4가지 태스크를 체계적으로 진행한다.

### 1.1 산출물 목표

| 산출물 | 형식 | 설명 |
|--------|------|------|
| 운영정책 결정서 | spec.md 3절 | 25개 정책 최종 결정 (추천안 검토 후 확정) |
| 기능 로드맵 | spec.md 4절 | 88개 기능 도메인별/우선순위별 최종 분류 |
| CUSTOM 모듈 정의서 | spec.md 5절 | 9개 모듈 기능 범위, 접근 전략, API 방향 |
| 후속 SPEC 계획 | spec.md 6절 | 7개 도메인별 SPEC ID, 범위, 접근 방식 |
| 아키텍처 결정 기록 | spec.md 7절 | Hybrid 전략, 기술 스택, 제약사항 |

---

## 2. 태스크 분해

### Task 1: 25개 운영정책 검토 및 확정

**목표**: 지니님이 25개 정책 추천안을 검토하고 최종 결정을 내린다.

**세부 단계**:

1. **런칭 필수 정책 14개 우선 검토**
   - 배송비 6개: 무료배송 기준, 배송비 금액, 배너 제외, 혼합주문, 제주, 도서산간
   - 결제 3개: PG사, 간편결제, PG 수수료
   - 쿠폰 3개: 신규가입, 동시사용, 유효기간
   - 리뷰 2개: 노출 방식, 작성 보상

2. **런칭 후 조정 정책 11개 검토**
   - 결제 2개: 결제 실패 안내, 무이자할부
   - 쿠폰 7개: 재구매/VIP/리뷰유도 쿠폰 금액 및 기준
   - 리뷰 2개: 삭제 시 회수, 사진 리뷰 추가 보상

3. **결정 방식**
   - 각 정책에 대해 '결정' 또는 '보류(사유 명시)' 선택
   - 추천안과 다른 결정 시 근거 기록
   - shopby 설정만으로 가능한 정책(10개)은 빠른 결정 가능

**선행 조건**: 없음 (즉시 착수 가능)
**결정 주체**: 지니님 (사업 의사결정)

---

### Task 2: 88개 기능 도메인 분류 최종 확정

**목표**: 88개 기능의 도메인 분류, 우선순위, shopby 분류를 최종 확인한다.

**세부 단계**:

1. **도메인 분류 검증**
   - 7개 도메인별 기능 배분이 적절한지 확인
   - 도메인 간 기능 이동이 필요한 항목 식별
   - 쇼핑몰(A)과 관리자(B)의 도메인 매핑 검증

2. **우선순위 최종 확인**
   - P1 53개: 런칭 필수 기능 누락/과잉 점검
   - P2 29개: 런칭 후 확장 기능 적절성 확인
   - P3 6개: 성장기 도입 기능 확인

3. **shopby 분류 최종 확인**
   - NATIVE 40개: shopby 기본 제공 확인
   - SKIN 24개: API + 스킨 커스텀 범위 확인
   - CUSTOM 22개: 자체 개발 범위 확인
   - EXTERNAL 2개: 외부 연동 범위 확인

**선행 조건**: 없음 (Task 1과 병행 가능)
**결정 주체**: 지니님 + 개발팀

---

### Task 3: 9개 CUSTOM 모듈 상세 정의서 작성

**목표**: 각 CUSTOM 모듈의 기능 범위, 기술 접근, API 방향을 구체화한다.

**세부 단계**:

1. **P1 모듈 5개 상세 정의 (런칭 필수)**
   - 인쇄 파일 업로드/검수 (XL): 검증 항목, PitStop 연동, 3단계 결과 표시
   - 출력상품 종속옵션 + 가격 엔진 (XL): 캐스케이딩 의존 드롭다운, 8종 가격 매트릭스
   - 옵션보관함/재주문 (L): 인쇄옵션 조합 저장, 파일 연결, 보관 기간
   - 프린팅머니 충전 (M): PG 결제 -> shopby 적립금 전환, 보너스 정책
   - 인쇄 주문상태 트래킹 (L): 8단계 커스텀 상태 코드, shopby 5단계 매핑

2. **P2 모듈 2개 개요 정의 (런칭 후)**
   - 원장관리 시스템 (XL): 온/오프라인 통합 원장, ERP 도입 시기 결정 필요
   - B2B 후불결제 (L): 미결제/결제완료 상태, 월별 정산

3. **P3 모듈 2개 개요 정의 (안정화 후)**
   - 체험단 관리 (L): 모집/신청/당첨/후기 관리
   - 팀별 통계 (M): 공정관리 연계, Excel 다운

**선행 조건**: Task 1의 결제 정책 결정 (PG사, 간편결제)
**결정 주체**: 개발팀 + 지니님

---

### Task 4: 7개 후속 SPEC 생성 계획 확정

**목표**: 각 도메인별 SPEC의 ID, 범위, 접근 방식, 전제 조건을 확정한다.

**세부 단계**:

1. **SPEC 범위 확정**
   - 각 SPEC이 10~18개 기능 범위 유지하는지 확인
   - SPEC 간 기능 중복/누락 점검
   - CUSTOM 기능이 포함된 SPEC의 접근 전략 확인

2. **생성 순서 확정**
   - Primary: SPEC-MEMBER-001, SPEC-ORDER-001, SPEC-PRODUCT-001
   - Secondary: SPEC-MYPAGE-001, SPEC-PAGE-001
   - Tertiary: SPEC-CS-001, SPEC-STATS-001

3. **전제 조건 매핑**
   - 각 SPEC의 시작 전 필요한 결정/계약 사항 확인
   - 의존 관계 정리 (SPEC 간, 외부 서비스 간)

**선행 조건**: Task 1, Task 2, Task 3 완료
**결정 주체**: 지니님 + 개발팀

---

### Task 5: 아키텍처 결정사항 문서화

**목표**: 이미 확정된 아키텍처 결정사항을 정리하고 후속 SPEC에서 참조 가능하도록 문서화한다.

**세부 단계**:

1. **Hybrid 전략 문서화**
   - 3-Tier 구조 (NATIVE/SKIN, CUSTOM, 관리자) 상세 설명
   - 각 Tier의 기술 스택, 스타일링, API 접근 방식

2. **설계 원칙 문서화**
   - PC-First vs Mobile-First 대상 화면 목록
   - 단일 페이지 옵션 폼 (option_NEW) 패턴 명세
   - CSS 오버라이드 관리 원칙 (@MX:WARN)

3. **기술 스택 목록 확정**
   - 프론트엔드: React 18 + Webpack 5 + CSS Variables + Tailwind CSS
   - 백엔드 (CUSTOM): NestJS BFF
   - 외부 서비스: AWS S3, PitStop, 카카오 알림톡, 팝빌, PG

**선행 조건**: 없음 (이미 확정된 사항 정리)
**결정 주체**: 개발팀

## 5. 의존성 관계

### 5.1 태스크 간 의존성

```
Task 1 (정책 결정) ─────────────┐
                                 ├──> Task 4 (후속 SPEC 계획)
Task 2 (기능 분류) ─────────────┤
                                 │
Task 3 (CUSTOM 정의) ───────────┘

Task 5 (아키텍처 문서화) ──> 독립 수행 가능
```

- Task 1, Task 2: 병행 수행 가능

- Task 4: Task 1, 2, 3 완료 후 수행
- Task 5: 독립적으로 수행 가능

```
SPEC-MEMBER-001 ──> SPEC-ORDER-001 (회원 인증 -> 주문)

SPEC-ORDER-001 ──> SPEC-STATS-001 (주문 데이터 -> 통계)

SPEC-PAGE-001, SPEC-CS-001: 상대적 독립
```

| AWS S3 | SPEC-ORDER-001 | 런칭 전 필수 (파일 저장) |

## 6. 접근 전략

### 6.1 NATIVE/SKIN 기능 (64개) - shopby API 기반

**접근 방식**: shopby API 문서 + 설정 화면 분석

1. shopby 관리자에서 설정 가능한 항목 확인
2. shopby API 응답 구조 분석 (인증 방식, 페이지네이션, 에러 처리)
3. Aurora Skin 컴포넌트 매핑 (기존 컴포넌트 활용 가능 여부)
4. 커스텀 스킨 필요 범위 정의 (CSS 오버라이드 vs 컴포넌트 신규 개발)

**핵심 참조 문서**: research-shopby-auth.md (shopby API 인증 분석)

### 6.2 CUSTOM 기능 (22개) - 인쇄 도메인 리서치 기반

**접근 방식**: 경쟁사 벤치마크 + 인쇄 도메인 전문 지식

1. 경쟁사 분석 (레드프린팅, 와우프레스, Vistaprint, MOO)
2. 인쇄 공정 전문 지식 적용 (용지/소재, 후가공, 가격 구조)
3. 독립 모듈 설계 (shopby 의존성 제로, NestJS BFF + React)
4. Tailwind CSS + Radix UI 기반 UI 구현

**핵심 참조 문서**: research-printing.md (인쇄 도메인 리서치)

### 6.3 EXTERNAL 기능 (2개) - 외부 API 문서 기반

**접근 방식**: 각 외부 서비스의 공식 API 문서 및 SDK 가이드

1. 구글/애플 OAuth: 각 플랫폼 공식 문서 기반 연동 (P2)
2. PG 결제: 이니시스 API 가이드 + shopby EP(External Payment) 연동 가이드

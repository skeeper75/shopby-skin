## 1. 개요

### 1.1 프로젝트 배경

후니프린팅(buysangsang.com)은 기존 자체 구축 사이트에서 **NHN Commerce shopby Enterprise** 플랫폼으로 이전하는 리뉴얼 프로젝트를 진행 중이다. 본 SPEC은 리뉴얼의 **마스터 기획서**로서, 화면설계(SPEC-SCREEN-001) 전 단계에서 필요한 모든 기획 결정사항을 취합하고 체계화한다.

### 1.2 목적

1. **운영정책 25개 결정**: 배송비/결제/쿠폰/리뷰 정책에 대한 벤치마크 기반 추천안 제시
2. **기능 로드맵 확정**: 88개 기능의 도메인 분류 및 P1/P2/P3 우선순위 체계 수립
3. **CUSTOM 모듈 정의**: shopby 미지원 9개 핵심 모듈의 기능 범위 및 접근 전략 명확화
4. **후속 SPEC 계획**: 7개 도메인별 SPEC 생성 로드맵 수립

### 1.3 범위

| 구분 | 포함 | 제외 |
|------|------|------|
| 기능 정의 | 88개 포함 기능의 요구사항 | 25개 제외 기능 (3.1절 참조) |
| 정책 결정 | 25개 운영정책 추천안 | 구체적 UI/UX 디자인 |
| 아키텍처 | Hybrid 전략, 기술 스택 확정 | 상세 DB 스키마, API 설계 |
| 로드맵 | 7개 도메인별 SPEC 계획 | 일정 예측, 인력 배치 |

## 2. EARS 요구사항

### Module 1: 운영정책 결정 체계

> 25개 운영정책에 대한 추천안 제시 및 결정 가이드 체계

**REQ-PLAN-001** [Ubiquitous]
시스템은 항상 25개 운영정책 각각에 대해 추천안, 선택지, 벤치마크 근거, shopby 설정 가능 여부를 포함한 결정 가이드를 제공해야 한다.

**REQ-PLAN-002** [Event-Driven]
WHEN 운영자가 특정 정책의 추천안을 검토하고 최종 결정을 입력하면, THEN 해당 정책의 상태가 '미결정'에서 '결정' 또는 '보류(사유 명시)'로 변경되어야 한다.

**REQ-PLAN-003** [State-Driven]
IF 25개 정책 중 미결정 항목이 존재하면, THEN 해당 정책과 연관된 화면설계 작업은 '대기' 상태로 표시되어야 한다.

**REQ-PLAN-004** [Ubiquitous]
시스템은 항상 정책을 4개 도메인(배송비 6개, 결제 5개, 쿠폰 10개, 리뷰 4개)으로 분류하고, 런칭 필수(14개)와 런칭 후 조정(11개)으로 구분해야 한다.

**REQ-PLAN-005** [Unwanted]
시스템은 벤치마크 근거 없이 운영정책 추천안을 제시하지 않아야 한다.

### Module 2: 기능 로드맵 및 도메인 분류

> 88개 기능의 P1/P2/P3 우선순위 체계 및 7개 도메인 구조

**REQ-PLAN-010** [Ubiquitous]
시스템은 항상 88개 기능을 shopby 분류(NATIVE 40개, SKIN 24개, CUSTOM 22개, EXTERNAL 2개)와 우선순위(P1 53개, P2 29개, P3 6개)로 체계적으로 분류해야 한다.

**REQ-PLAN-011** [Ubiquitous]
시스템은 항상 88개 기능을 7개 도메인으로 그룹화하여 관리해야 한다:
- A1A2-MEMBER (로그인/회원): 18개
- A3-MYPAGE (마이페이지): 16개
- A10B4-PRODUCT (상품관리): 14개
- A6B8-ORDER (주문관리): 15개
- A4B5-CS + B1-ADMIN (고객센터/관리자): 9개
- Pages + A7A8-CONTENT + A5-PAYMENT (페이지/콘텐츠/결제): 9개
- B7-STATISTICS (통계): 7개

**REQ-PLAN-012** [Event-Driven]
WHEN 특정 기능의 key_decision이 결정되면, THEN 해당 기능의 화면설계 입력 정보 완성도가 갱신되어야 한다.

**REQ-PLAN-013** [State-Driven]
IF 특정 기능이 CUSTOM으로 분류되어 있으면, THEN 해당 기능의 기획은 인쇄 도메인 리서치 및 경쟁사 벤치마크에 기반해야 한다 (shopby 문서가 아닌 별도 접근).

**REQ-PLAN-014** [State-Driven]
IF 특정 기능이 NATIVE 또는 SKIN으로 분류되어 있으면, THEN 해당 기능의 기획은 shopby API 문서 및 설정 가능 여부에 기반해야 한다.

### Module 3: CUSTOM 개발 모듈 정의

> 9개 핵심 자체 개발 모듈의 기능 범위 및 접근 전략

**REQ-PLAN-020** [Ubiquitous]
시스템은 항상 9개 CUSTOM 모듈 각각에 대해 기능 범위, 기술 접근, API 방향, 의존성을 문서화해야 한다.

**REQ-PLAN-021** [Ubiquitous]
시스템은 항상 CUSTOM 모듈의 기획을 shopby 외부의 인쇄 도메인 전문 리서치에 기반해야 한다.

**REQ-PLAN-022** [Event-Driven]
WHEN CUSTOM 모듈의 상세 정의가 완료되면, THEN 해당 모듈에 대한 독립 SPEC이 생성 가능한 상태가 되어야 한다.

**REQ-PLAN-023** [State-Driven]
IF CUSTOM 모듈이 P1 우선순위이면, THEN 해당 모듈은 런칭 필수 기능으로 분류되고 후속 SPEC에서 우선 구현되어야 한다.

**REQ-PLAN-024** [Unwanted]
시스템은 CUSTOM 모듈의 기능 범위를 shopby 문서에만 의존하여 정의하지 않아야 한다. 인쇄 도메인 특화 리서치가 반드시 포함되어야 한다.

---

### Module 4: 아키텍처 결정사항

> Hybrid 전략, 기술 스택, 설계 원칙 확정

**REQ-PLAN-030** [Ubiquitous]
시스템은 항상 다음 3-Tier Hybrid 아키텍처를 따라야 한다:
- Tier 1 (NATIVE/SKIN): Aurora Skin 기반 shopby Provider + API 래퍼
- Tier 2 (CUSTOM): 독립 모듈, Tailwind CSS + Radix UI, NestJS BFF
- Tier 3 (관리자): Tailwind CSS + shadcn/ui, AdminLayout 독자 체계

**REQ-PLAN-031** [State-Driven]
IF 기능이 인쇄 특화 화면(상품옵션, 파일 업로드, 편집 미리보기, 가격 시뮬레이터, 관리자 전체)이면, THEN PC-First 설계를 적용해야 한다.

**REQ-PLAN-032** [State-Driven]
IF 기능이 일반 쇼핑몰 화면(로그인, 회원가입, 마이페이지, 장바구니, 결제)이면, THEN Mobile-First 설계를 적용해야 한다.

**REQ-PLAN-033** [Ubiquitous]
시스템은 항상 상품 옵션 선택에 단일 페이지 스크롤 폼(피그마 option_NEW 패턴)을 사용해야 한다. Step Wizard 방식은 사용하지 않는다.

**REQ-PLAN-034** [Ubiquitous]
시스템은 항상 확정된 기술 스택(React 18 + Webpack 5 + Tailwind CSS + NestJS BFF)을 준수해야 한다.

### Module 5: 후속 SPEC 생성 계획

> 7개 도메인별 SPEC 생성 로드맵

**REQ-PLAN-040** [Event-Driven]
WHEN 25개 운영정책 결정이 완료되면, THEN 7개 도메인별 후속 SPEC 생성이 착수되어야 한다.

**REQ-PLAN-041** [Ubiquitous]
시스템은 항상 각 후속 SPEC이 10~18개 기능 범위 내에 있도록 도메인을 분할해야 한다.

**REQ-PLAN-042** [State-Driven]
IF 후속 SPEC에 CUSTOM 기능이 포함되어 있으면, THEN 해당 SPEC은 인쇄 도메인 리서치 기반 접근 전략을 명시해야 한다.

**REQ-PLAN-043** [State-Driven]
IF 후속 SPEC에 NATIVE/SKIN 기능만 포함되어 있으면, THEN 해당 SPEC은 shopby API 기반 접근 전략을 명시해야 한다.

**REQ-PLAN-044** [Unwanted]
시스템은 운영정책 미결정 상태에서 후속 SPEC의 화면설계를 시작하지 않아야 한다.

## 4. 88개 기능 도메인별 분류 현황

| 분류 | 수량 | 비율 | 설명 |

| NATIVE | 40개 | 45% | shopby 기본 제공, 스킨 래핑 또는 설정만 필요 |
| SKIN | 24개 | 27% | shopby API + 커스텀 UI(스킨) 구현 |

### 4.2 우선순위별 통계

| 항목 | 확정 기술 | 제약 사항 |

### 7.4 제외된 25개 기능

shopby 기본 기능으로 충분하거나 2차 개발 범위로 이관된 25개 기능은 본 SPEC 범위에서 제외한다.

- 쇼핑몰 측 제외 12개: 옵션보관함 내역, 나의활동, 상품Q&A, 이용후기 메인, 체험단 모집/활동내역, 대량주문 견적, 기업인쇄상담, 디자인상담, 출력상품 주문하기 4종, 포장재/굿즈/수작 주문, 랜딩페이지 5종
- 관리자 측 제외 13개: 기업인쇄/디자인 상담 관리, 디자인의뢰, 거래처/계좌/원장/미수금 관리, 상품Q&A 관리, shopby 기본 설정 4종
